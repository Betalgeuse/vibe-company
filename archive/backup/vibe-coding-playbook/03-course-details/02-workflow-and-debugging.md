## Debugging 원칙

### 1. 인간의 감에 의존하지 말 것

**금지:**
- "아마 이게 문제일 것 같은데..."
- "대충 이 부분이 이상해 보이는데..."

**권장:**
- Comet/Codex 등 도구의 로그 분석 결과 신뢰
- 데이터와 로그가 말하게 하라
- 에러 메시지를 정확히 읽기

### 비유

```
의사의 진단

❌ 나쁜 의사: "음... 열나니까 감기같은데요?"
✅ 좋은 의사: "체온 38.5도, 혈액 검사 결과 백혈구 증가..."
```

### 2. 디버깅 중 대기 시간 활용

**상황**: 디버깅 도구가 실행되는 동안 기다리는 시간

**해결**: 화면 분할로 병렬 작업

```bash
# Pane 1: 디버깅 실행
npm run debug

# Pane 2: 로그 모니터링
tail -f debug.log

# Pane 3: 대기 시간 동안 문서 작업
vim README.md
```

### 3. Context Switching 금지

**안티패턴:**
- 터미널 창 하나만 띄우고 멍하니 기다리기
- 디버깅 끝날 때까지 아무것도 안 하기

**권장:**
- 디버깅이 돌아가는 동안 다른 Pane에서 병렬 작업
- 문서 작성, 테스트 코드 작성 등

## 시간 절약 팁

### Tip 1: 단축키 마스터

자주 쓰는 명령은 단축키로:
```bash
# .bashrc or .zshrc
alias gs='git status'
alias gp='git push'
alias gd='git diff'
alias nr='npm run'
```

### Tip 2: 스니펫 활용

반복되는 코드는 스니펫으로:
- VS Code Snippets
- Vim UltiSnips
- TextExpander

### Tip 3: 자동화

반복 작업은 스크립트로:
```bash
# auto-test.sh
npm run test
npm run lint
npm run build
```

## 흔한 실수와 해결

### 실수 1: Reference 안 찾고 바로 코딩

**문제**: 방향성 없이 헤매게 됨

**해결**: 10분 투자해서 Reference 3개 찾기

**문제**: 시간 낭비

**해결**: 화면 분할해서 병렬 작업

### 실수 3: 에러 메시지 안 읽기

**문제**: 같은 에러 반복

**해결**: 에러 메시지를 정확히 읽고 Google 검색
