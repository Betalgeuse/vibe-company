# Checklist: Reference 기반 개발

> ⚠️ **코딩 시작 전 확인사항** - 레퍼런스 없이 시작하지 마세요!

## ✅ 작업 시작 전

### 1. 레퍼런스 3개 찾기
```bash
# 검색 예시
"best practice [기능명]"
"[기능명] example site:github.com"
"how [큰 회사] built [기능명]"
```
- [ ] 글로벌 Top 서비스 예시 1개+
- [ ] GitHub Star 1000+ 프로젝트 1개+
- [ ] 공식 문서 예시 1개+

### 2. 레퍼런스 품질 확인
- [ ] 최신인가? (1년 이내)
- [ ] 프로덕션에서 사용 중인가?
- [ ] 내 상황과 비슷한가?

### 3. 패턴 추출
- [ ] 공통 패턴 파악했는가?
- [ ] 스크린샷/코드 저장했는가?
- [ ] 차이점을 이해했는가?

---

## ✅ AI 명령 시

### 4. 레퍼런스 첨부 확인
```
❌ 나쁜 명령:
"로그인 기능 만들어줘"

✅ 좋은 명령:
"Vercel (https://vercel.com/login)처럼 로그인 만들어줘.
 - Magic link 방식
 - 이메일만 입력
 - Supabase Auth 사용"
```
- [ ] 레퍼런스 URL 첨부했는가?
- [ ] 구체적 요구사항 명시했는가?
- [ ] 사용할 라이브러리 지정했는가?

### 5. Context 제공
- [ ] 현재 tech stack 명시
- [ ] 기존 코드 구조 설명
- [ ] 제약사항 명시

---

## ✅ 실행 후 검증

### 6. 레퍼런스와 비교
- [ ] 핵심 패턴이 적용되었는가?
- [ ] 빠진 부분은 없는가?
- [ ] 보안/성능은 고려되었는가?

### 7. FCSR 기록
```
오늘의 작업: [작업명]
레퍼런스: [URL 1], [URL 2], [URL 3]
첫 시도 성공: ✅ / ❌
재시도 횟수: [N]회
```
- [ ] 첫 시도에 성공했는가?
- [ ] 실패 시 원인 파악했는가?
- [ ] 다음에는 어떻게 개선할지 메모했는가?

---

## 🚨 트러블슈팅

### ❌ "레퍼런스를 못 찾겠어요"
**해결:**
```
1. 검색어를 영어로 바꾸기
2. 기능을 더 작은 단위로 쪼개기
3. 비슷한 기능으로 확장해서 찾기
   예: "실시간 채팅" 못 찾음
    → "websocket example" 검색
```

### ❌ "레퍼런스가 너무 복잡해요"
**해결:**
```
1. 핵심 부분만 추출
2. 단순화된 버전 먼저 구현
3. 단계적으로 기능 추가
```

### ❌ "레퍼런스와 다른 결과가 나와요"
**해결:**
```
1. Tech stack 차이 확인
2. 버전 차이 확인
3. 레퍼런스 코드를 그대로 복사해서 테스트
4. 한 줄씩 우리 코드로 변경
```
