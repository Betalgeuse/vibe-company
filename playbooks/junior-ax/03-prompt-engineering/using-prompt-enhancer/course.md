# 📚 Skill 사용하기 - Course

## 개요

**Skill**은 Factory에서 제공하는 재사용 가능한 작업 자동화 도구입니다.
복잡한 작업을 간단한 명령 하나로 실행할 수 있습니다.

---

## 1. Skill이란?

### Droid vs Skill

| 구분 | Droid | Skill |
|------|-------|-------|
| **용도** | 특정 도메인 전문가 역할 | 작업 패턴 자동화 |
| **호출 방식** | Task tool로 호출 | Skill tool로 호출 |
| **상태** | Stateless | 대화 내에서 동작 |
| **예시** | `@b2b-market-hunter` | `prompt-enhancer` |

### 사용 가능한 Skill 목록

현재 `.factory/skills/`에 있는 skill:

| Skill | 용도 |
|-------|------|
| `prompt-enhancer` | 간단한 프롬프트를 구조화된 요청으로 변환 |
| `codex-architecture-review` | Codex CLI로 아키텍처 리뷰 진행 |

## 2. prompt-enhancer Skill 사용법

### 기본 사용 패턴

**Before (막연한 요청):**
```
로그인 기능 추가해줘
```

**With Skill:**
```
간단한 프롬프트 + prompt-enhancer skill을 사용해서 내 요청사항을 구조화해줘:

로그인 기능 추가해줘
```

**After (구조화된 요청):**
```markdown
<문제 정의>
현재 애플리케이션에 인증 시스템이 없어서 사용자 구분이 불가능합니다.
</문제 정의>

<목표>
- [ ] 이메일/비밀번호 기반 로그인 구현
- [ ] JWT 토큰 인증
- [ ] Protected Route 설정
</목표>

<비목표>
- 기존 UI 변경하지 않음
- 회원가입 기능 제외
- 소셜 로그인 제외
</비목표>

<성공 기준>
- 올바른 credentials로 로그인 시 JWT 토큰 발급
- Protected routes 접근 제어 동작
- 새로고침 후에도 로그인 상태 유지
</성공 기준>

<실행 계획>
1. Backend: JWT 인증 미들웨어 구현
2. Frontend: AuthContext + Login 컴포넌트
3. ProtectedRoute 래퍼 컴포넌트 생성
...
</실행 계획>
```

### 실제 사용 예시

#### 예시 1: 기능 개발 요청

**입력:**
```
prompt-enhancer skill 써서 이거 구조화해줘:

대시보드에 차트 추가하고 싶어
```

**결과:** Skill이 질문을 통해 정보를 수집하고 구조화된 프롬프트 생성

#### 예시 2: 버그 수정 요청

**입력:**
```
prompt-enhancer skill로 구조화:

버튼 클릭이 안 돼
```

**결과:**
```markdown
<문제 정의>
특정 버튼 클릭 시 이벤트가 발생하지 않음
</문제 정의>

<목표>
- [ ] 버튼 클릭 이벤트 핸들러 정상 동작
</목표>

<비목표>
- 버튼 스타일 변경 제외
- 다른 UI 요소 수정 제외
</비목표>
...
```

## 3. 핵심 원칙: <비목표> 섹션

**가장 중요한 원칙:**

> **"목표를 달성하기 위해 굳이 불필요한 UI를 변경하거나 수정하지 않는다"**

### 왜 중요한가?

1. **범위 통제**: AI가 요청 이상의 작업을 하지 않도록 방지
2. **예측 가능성**: 무엇이 변경될지 명확히 알 수 있음
3. **리뷰 용이**: 변경 범위가 명확하면 코드 리뷰가 쉬움
4. **롤백 가능**: 문제 발생 시 영향 범위 파악 용이

### 비목표 작성 예시

```markdown
<비목표>
- 기존 동작하는 코드 변경 최소화
- 백엔드 로직 수정 시 프론트엔드 UI 변경 금지
- 리팩토링은 필요한 경우에만 최소한으로
- 요청 범위 외 추가 기능 개발 금지
</비목표>
```

```
┌─────────────────────────────────────────┐
│  1. 간단한 요청 작성                     │
│     "로그인 기능 추가해줘"               │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│  2. Skill 호출                          │
│     "prompt-enhancer skill 써서         │
│      내 요청 구조화해줘"                 │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│  3. Skill이 질문 (필요시)               │
│     "어떤 인증 방식을 원하시나요?"       │
│     "UI도 수정해야 하나요?"              │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│  4. 구조화된 프롬프트 생성               │
│     <문제정의> <목표> <비목표>           │
│     <성공기준> <실행계획>                │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│  5. 검토 및 승인                        │
│     "이 계획으로 진행할까요?"            │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│  6. 실행                                │
│     AI가 계획대로 구현                   │
└─────────────────────────────────────────┘
```

## 5. 프롬프트 템플릿 구조

prompt-enhancer가 생성하는 구조:

```markdown
<문제 정의>
[현재 무엇이 문제인지 - 구체적으로]
[Why is this a problem? 영향은?]
</문제 정의>

<목표>
- [ ] 구현할 기능 1 (구체적으로)
- [ ] 구현할 기능 2 (구체적으로)
</목표>

<비목표>
- 불필요한 UI 변경 금지
- [이 작업에서 하지 않을 것들]
- [범위를 벗어나는 것들]
</비목표>

<성공 기준>
- [완료 상태 정의 - 측정 가능하게]
- [테스트 방법]
- [예상 결과물]
</성공 기준>

<실행 계획>
1. [단계 1 - 구체적 작업]
   - 어떤 파일을 수정할 것인지
   - 예상되는 영향 범위
2. [단계 2 - 구체적 작업]
...
</실행 계획>
```

## 6. 언제 Skill을 사용해야 하는가?

### ✅ 사용하면 좋은 경우

- 기능 개발/구현 요청
- 복잡한 버그 수정
- 여러 파일에 걸친 작업
- 명확한 범위 설정이 필요한 경우
- FCSR을 높이고 싶을 때

### ❌ 사용하지 않아도 되는 경우

- 간단한 질문 ("이 코드가 뭐하는 거야?")
- 이미 구조화된 요청
- 한 줄 수정 수준의 작업
- 탐색/조사 작업

## 7. 실전 팁

### Tip 1: 비목표를 먼저 생각하라

요청 전에 "이건 하면 안 돼"를 먼저 정의하면 범위가 명확해집니다.

### Tip 2: 성공 기준은 테스트 가능하게

"잘 동작해야 함" ❌
"로그인 버튼 클릭 시 JWT 토큰이 localStorage에 저장됨" ✅

### Tip 3: Plan Mode 활용

Skill은 바로 실행하지 않고 먼저 계획을 보여줍니다.
반드시 검토 후 "진행해줘"라고 승인하세요.
