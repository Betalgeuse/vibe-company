## 🎯 The One Number: CFS (Customer Fit Score)

호텔의 ADR(객단가)처럼, 고객 선별의 성공을 측정할 단 하나의 지표.

**[VIDEO: CFS 개념 소개 - 왜 모든 고객을 다 받으면 안 되는가?]**

### 정의 (쉽게 풀어쓰면)

**Customer Fit Score (고객 적합도 점수)**

미팅한 고객이 우리의 ICP(Ideal Customer Profile)에 얼마나 부합하는지를 0~100점으로 평가한 점수입니다.

**비유로 이해하기:**
레스토랑에서 손님을 고르는 것과 같습니다.
- ❌ 나쁜 예: "손님이면 다 받자! 매출이 중요해" → 취객, 문제 손님까지 다 받아서 평판 하락
- ✅ 좋은 예: "우리 레스토랑 컨셉에 맞는 손님만 받자" → 만족도 높은 단골 고객 확보

### 왜 이 숫자가 중요한가?

**1. 리소스 낭비를 막습니다**
- 잘못된 고객과 일하면 몇 달이 날아갑니다
- 제대로 된 고객 한 명이 잘못된 고객 열 명보다 낫습니다

**2. 팀의 사기를 지킵니다**
- 끝없는 요구사항 변경과 불만족한 고객 → 팀 번아웃
- 문제를 이해하는 고객 → 팀의 성취감과 성장

**3. 회사의 평판을 만듭니다**
- 성공 사례 = 다음 고객을 부릅니다
- 실패 사례 = 시장에서의 신뢰 하락

### 계산 방식 (5가지 기준)

```
CFS = (기준 1 + 기준 2 + 기준 3 + 기준 4 + 기준 5) ÷ 5

각 기준은 0~20점 (총점 100점)

기준 1: AI 시도 경험 (Tried but Failed 점수)
기준 2: 의사결정권자 접근성 (Decision Maker Access)
기준 3: 문제의 명확성 (Problem Clarity)
기준 4: 예산 현실성 (Budget Reality)
기준 5: 기술 성숙도 (Tech Maturity)
```

**시각화:**
```
┌─────────────────────────────────────────────────┐
│            CFS 점수별 액션                        │
├─────────────────────────────────────────────────┤
│                                                 │
│  90~100점: 🔥 최우선 고객 (Tier 1)               │
│            → 즉시 제안서 작성, CEO 직접 미팅      │
│                                                 │
│  70~89점: ⭐ 주요 고객 (Tier 2)                  │
│            → 1주일 내 제안서, 정기 팔로업         │
│                                                 │
│  50~69점: 🤔 보류 (Watch & Wait)                │
│            → 6개월 후 재접촉, 니즈 변화 대기      │
│                                                 │
│  50점 미만: ❌ 거절 (Polite Decline)             │
│            → 정중히 거절, 다른 솔루션 추천        │
│                                                 │
└─────────────────────────────────────────────────┘
```

## 📘 The 4C Framework

고객 선별의 핵심을 4가지 요소로 구조화합니다.

**비유:** 의사가 환자를 진단하는 과정과 같습니다.
- **Camcorder (관찰)**: 환자의 증상을 자세히 관찰
- **Course (진단)**: 체계적인 문진과 검사
- **Cadence (주기)**: 정기 검진 스케줄
- **Checklist (판단)**: 치료 가능 여부 최종 판단

```
┌─────────────────────────────────────────────────┐
│      Customer Qualification 4C Framework        │
├─────────────────────────────────────────────────┤
│                                                 │
│  1. 🎥 Camcorder  → 고객 미팅 녹화               │
│     "이런 신호를 주의 깊게 들어라"                │
│                                                 │
│  2. 📚 Course     → 단계별 검증 프로세스         │
│     "이 순서대로 질문하고 판단하라"              │
│                                                 │
│  3. ⏰ Cadence    → 평가 타이밍                  │
│     "언제 다시 확인하고 판단할 것인가"           │
│                                                 │
│  4. ✅ Checklist  → 최종 의사결정                │
│     "받을 것인가, 거절할 것인가"                 │
│                                                 │
└─────────────────────────────────────────────────┘
```

## 1. 🎥 Camcorder (녹화해야 할 핵심 고객 미팅)

고객의 말과 행동에서 성공 신호와 위험 신호를 포착하는 방법을 녹화합니다.

**왜 녹화인가?**
- 말로만 듣는 것보다 실제 미팅 영상을 보면 놓친 신호를 발견합니다
- 팀원들이 "아, 이런 신호가 위험 신호였구나"를 학습합니다
- 나중에 비슷한 고객을 만났을 때 빠르게 판단할 수 있습니다

### Meeting 1: The First Discovery Call

**목적:** 30분 안에 이 고객이 "Tried but Failed" 고객인지 판단하기

**쉽게 설명하면:**
첫 데이트에서 상대방이 진심인지 아닌지를 파악하는 것과 같습니다.
말뿐인 사람은 시간 낭비, 진심인 사람은 깊이 알아가야죠.

**[VIDEO: 첫 미팅에서 꼭 포착해야 할 5가지 신호]**

**녹화해야 할 핵심 순간들:**

#### 순간 1: 자기소개 30초 (고객이 자신을 어떻게 소개하는가)

**성공 신호 ✅:**
```
고객: "저희는 작년에 자체적으로 AI 챗봇을 도입해봤는데요, 
      hallucination 문제로 3개월 만에 중단했습니다. 
      그 과정에서 2억 정도를 썼고, 이번에는 제대로 하고 싶어서 
      전문가를 찾고 있습니다."
```

**위험 신호 ❌:**
```
고객: "AI가 요즘 핫하잖아요? 우리도 한번 해볼까 해서요. 
      근데 돈은 별로 없어요. 빠르게 좀 해주실 수 있어요?"
```

**왜 다른가?**
- ✅ 성공 신호: 구체적인 실패 경험 + 학습 + 예산 준비 + 진지함
- ❌ 위험 신호: 모호함 + 트렌드 추종 + 예산 없음 + 빠른 결과 기대

#### 순간 2: 문제 설명 2분 (어떤 문제를 풀고 싶은가)

**성공 신호 ✅:**
```
고객: "저희 CS팀이 하루에 1,000건의 문의를 처리하는데, 
      그 중 70%가 반복 질문입니다. 
      현재 5명의 직원이 이 업무만 하고 있고, 
      월 인건비가 2천만 원 정도 나갑니다. 
      이걸 AI로 자동화하면 3명은 다른 일을 할 수 있을 것 같습니다."
```

**위험 신호 ❌:**
```
고객: "저희도 AI 좀 넣어서... 뭔가 효율화하고 싶어요. 
      다른 회사들도 다 하던데... 
      어떤 걸 할 수 있는지 제안 좀 해주시겠어요?"
```

**핵심 차이:**
```
┌──────────────────────────────────────────────┐
│  성공 신호              위험 신호              │
├──────────────────────────────────────────────┤
│  구체적 숫자           모호한 표현             │
│  명확한 문제           문제를 모름             │
│  측정 가능             측정 불가능             │
│  솔루션 방향 있음      솔루션 요구             │
└──────────────────────────────────────────────┘
```

#### 순간 3: 과거 시도 경험 3분 (무엇을 시도했고 왜 실패했는가)

**최고의 신호 🔥:**
```
고객: "6개월 전에 GPT API를 직접 연동해봤는데, 
      회사 내부 데이터와 연결이 안 되더라고요. 
      RAG도 시도했는데 retrieval 정확도가 60%밖에 안 나왔습니다. 
      결국 우리가 ML/AI 전문가가 아니라는 걸 깨달았고, 
      이제는 제대로 된 아키텍처가 필요하다고 판단했습니다."
```

**주의 신호 ⚠️:**
```
고객: "작년에 다른 AI 회사랑 했는데 별로였어요. 
      가격만 비싸고 제대로 안 되더라고요. 
      이번에는 제대로 해주시는 거죠?"
```

**분석:**
- 🔥 최고: 기술 용어 사용 + 구체적 실패 경험 + 원인 분석 + 해결책 인식
- ⚠️ 주의: 타사 비난 + 모호한 평가 + 우리에게 책임 전가

**녹화 시점:** 모든 첫 미팅 (필수)

**팀 리뷰 포인트:**
```
미팅 후 1시간 이내에 팀 디브리핑
- 성공 신호 몇 개 나왔는가?
- 위험 신호 몇 개 나왔는가?
- CFS 점수는 몇 점인가?
- 다음 미팅 진행 여부 결정
```


### Meeting 2: Technical Deep Dive

**목적:** 고객의 기술 인프라와 데이터 성숙도 파악하기

**쉽게 설명하면:**
집을 리모델링하기 전에 건축 구조를 확인하는 것과 같습니다.
기초가 튼튼하면 빠르게 진행, 기초가 약하면 추가 작업 필요.

**[VIDEO: 기술 미팅에서 확인해야 할 체크리스트]**

**녹화해야 할 핵심 순간들:**

#### 순간 1: 데이터 인프라 확인

**성공 신호 ✅:**
```
고객: "저희는 AWS를 쓰고 있고, RDS에 고객 데이터가 있습니다. 
      S3에는 과거 3년치 로그가 저장되어 있고요. 
      Redshift로 분석도 하고 있습니다. 
      API 문서도 준비되어 있어서 공유해드릴 수 있습니다."
```

**위험 신호 ❌:**
```
고객: "데이터요? 엑셀에 있어요. 
      서버는... IT 팀에 물어봐야 할 것 같은데요. 
      API는 뭔지 잘 모르겠고..."
```

#### 순간 2: 의사결정 프로세스

**성공 신호 ✅:**
```
고객: "저는 CTO이고, 이 프로젝트에 대한 결재 권한이 있습니다. 
      제안서 받으면 2주 안에 내부 검토 후 답변 드릴 수 있습니다. 
      예산은 이미 확보되어 있고, 올해 Q2에 시작하려고 합니다."
```

**위험 신호 ❌:**
```
고객: "일단 제안서 받아서 위에 올려봐야 해요. 
      언제 결정날지는... 잘 모르겠네요. 
      예산은 아직 확정 안 됐고요."
```

**녹화 시점:** ICP 부합 고객과의 2차 미팅

**핵심 판단 기준:**
```
┌────────────────────────────────────────┐
│  기술 성숙도 레벨                       │
├────────────────────────────────────────┤
│  Level 4: 클라우드 + API + 데이터 정책  │
│          → 즉시 프로젝트 진행 가능      │
│                                        │
│  Level 3: 클라우드 + 데이터베이스       │
│          → 1개월 준비 후 진행 가능      │
│                                        │
│  Level 2: 온프레미스 서버 + 엑셀        │
│          → 3개월 기초 작업 필요         │
│                                        │
│  Level 1: 데이터 없음 / 엑셀만          │
│          → 6개월+ 또는 프로젝트 불가    │
└────────────────────────────────────────┘

최소 Level 2 이상만 진행!
```


### Meeting 3: Budget & Timeline Alignment

**목적:** 예산과 기대치가 현실적인지 확인하기

**녹화해야 할 핵심 순간들:**

```
고객: "이번 프로젝트에 2억 정도 책정했습니다.
```

```
고객: "예산이요? 일단 얼마나 드는지 알려주세요.

      최대한 저렴하게 해주실 수 있죠?"
```

#### 순간 2: 성공 기준 논의

**명확한 신호 ✅:**
```
고객: "성공 기준은 3가지입니다:
      1) CS 응답 시간 50% 단축
      2) 고객 만족도 80점 이상 유지
      3) 월 운영 비용 1천만 원 이하
      이 세 가지가 충족되면 성공으로 봅니다."
```

**모호한 신호 ❌:**
```
고객: "그냥... 잘 되면 되는 거 아닌가요? 
      AI가 알아서 잘 해주면 되죠. 
      기대치요? 글쎄요... 좋아지면 좋은 거죠."
```

**핵심 원칙:**
> 예산과 성공 기준이 명확하지 않은 고객 = 나중에 100% 불만족  
> 처음부터 명확하게 정의하지 않으면 무조건 문제 발생!

#### Vibers의 ICP: "Tried but Failed" MBA 출신 구매자

**핵심 특징:**

```
┌─────────────────────────────────────────────────┐
│    "Tried but Failed" 고객의 3가지 특징          │
├─────────────────────────────────────────────────┤
│                                                 │
│  1. 🔥 AI 도입 시도 & 실패 경험                  │
│     → 최소 1회 이상의 AI 프로젝트 실패          │
│     → 실패 원인을 분석하고 이해함                │
│     → "왜 안 됐는지" 말할 수 있음                │
│                                                 │
│  2. 💰 예산 확보 & 의사결정권                    │
│     → 최소 1억 이상의 프로젝트 예산 확보         │
│     → C-level 또는 VP급 의사결정권자            │
│     → 빠른 의사결정 가능 (2주 이내)              │
│                                                 │
│  3. 🎯 명확한 비즈니스 문제 인식                 │
│     → "AI를 하고 싶다" ❌                        │
│     → "이 문제를 AI로 풀고 싶다" ✅              │
│     → ROI 계산이 가능한 구체적 문제              │
│                                                 │
└─────────────────────────────────────────────────┘
```

#### ICP 점수 계산법

**기준 1: AI 시도 경험 (0~20점)**

```
20점: 자체 AI 프로젝트 진행 → 실패 → 원인 분석 완료
15점: AI 솔루션 도입 → 실패 → 문제 인식
10점: AI POC 진행 → 중단 → 막연한 실패감
5점:  AI 도입 검토만 함, 실행 안 함
0점:  AI 경험 전혀 없음
```

**기준 2: 의사결정권자 접근성 (0~20점)**

```
20점: 미팅 상대 = C-level, 즉시 결정 가능
15점: 미팅 상대 = VP급, 1주일 내 결정
10점: 미팅 상대 = 팀장급, C-level 보고 필요
5점:  미팅 상대 = 담당자, 의사결정자 멀리 있음
0점:  의사결정 프로세스 불명확
```

**기준 3: 문제의 명확성 (0~20점)**

```
20점: 구체적 숫자로 문제 설명 가능
      예) "CS 처리 시간 평균 15분, 하루 1,000건"
15점: 정성적이지만 명확한 문제
      예) "고객 응대가 너무 느림"
10점: 문제는 있으나 측정 안 됨
      예) "뭔가 비효율적인 것 같음"
5점:  문제를 찾아달라고 요청
      예) "우리 회사에 AI 적용할 곳 찾아주세요"
0점:  문제 없음, 트렌드만 따라가려 함
```

**기준 4: 예산 현실성 (0~20점)**

```
20점: 예산 확보 완료, 금액 공개, 계약 준비 완료
15점: 예산 확보 완료, 금액 협의 가능
10점: 예산 책정 중, 승인 대기
5점:  예산 미정, "일단 제안서 보고 결정"
0점:  예산 없음, "최대한 저렴하게"
```

**기준 5: 기술 성숙도 (0~20점)**

```
20점: 클라우드 인프라 + API + 데이터 정책
15점: 클라우드 인프라 + 데이터베이스
10점: 온프레미스 서버 + 데이터베이스
5점:  데이터가 엑셀/스프레드시트에만 있음
0점:  데이터 없음, 수집부터 필요
```

#### Anti-ICP: 거절해야 하는 고객 유형

**유형 1: "트렌드 추종자" (Trend Chaser)**

```
특징:
- "AI가 핫하니까 우리도 해야죠"
- 구체적 문제 없음, 남들이 하니까 함
- 빠른 결과, 낮은 비용 기대
- 실패하면 "AI는 별로"라고 결론

위험:
→ 성공 기준 불명확으로 영원한 불만족
→ 프로젝트 중간에 관심 잃고 중단
→ 나쁜 평판 생성 (우리 탓으로 돌림)

대응:
✅ "구체적으로 어떤 문제를 해결하고 싶으신가요?"
✅ "성공을 어떻게 측정하실 건가요?"
✅ 명확한 답변 없으면 → 정중히 거절
```

**유형 2: "바겐 헌터" (Bargain Hunter)**

```
특징:
- "다른 데는 더 저렴하던데요"
- "일단 싸게 해보고 잘되면 더 할게요"
- 가격만 중요, 품질은 관심 없음
- 무료 컨설팅만 받으려 함

위험:
→ 프로젝트 중간에 예산 없다고 중단
→ 끝없는 가격 네고
→ 낮은 마진, 높은 스트레스

대응:
✅ "저희는 가장 저렴한 솔루션이 아닙니다"
✅ "하지만 가장 제대로 된 솔루션입니다"
✅ 가격에만 집착하면 → 다른 업체 추천
```

**유형 3: "책임 회피자" (Blame Shifter)**

```
특징:
- "전에 다른 회사가 망쳤어요"
- "이번엔 제대로 해주시는 거죠?"
- 자기 책임은 0%, 전부 파트너 탓
- 과거 실패를 배운 게 없음

위험:
→ 이번에도 실패하면 우리 탓
→ 내부 협조 안 됨, 우리만 고생
→ 성공해도 자기 공으로 돌림

대응:
✅ "과거 프로젝트가 실패한 이유를 어떻게 보시나요?"
✅ "이번에는 무엇을 다르게 하실 건가요?"
✅ 계속 남 탓만 하면 → 거절
```

**유형 4: "몽상가" (Dreamer)**

```
특징:
- "AGI처럼 모든 걸 다 하는 AI 만들어주세요"
- "비용은 얼마 안 들죠? AI니까"
- "한 달이면 되죠?"
- 현실감 제로

위험:
→ 불가능한 기대치
→ 기술적 한계 설명해도 이해 못 함
→ 실패 확정

대응:
✅ "그 범위라면 최소 6개월, 5억 필요합니다"
✅ "단계별로 진행하시는 게 현실적입니다"
✅ 계속 비현실적이면 → 정중히 거절
```

### Chapter 3: 문제의 본질 파악 (Problem Deep Dive)

#### AX 실패의 3가지 유형으로 분류하기

Vibers가 해결할 수 있는 문제는 딱 3가지입니다:

**유형 1: 비즈니스 맥락 이해 부재 (Hallucination Problem)**

```
증상:
- AI가 틀린 정보를 생성함
- 회사 규정과 다른 답변을 함
- 복잡한 비즈니스 룰을 이해 못 함

고객 예시:
"저희 금융상품은 100가지인데, 
 AI가 없는 상품을 추천하거나 
 잘못된 수수료를 안내합니다."

우리 솔루션:
→ 온톨로지 기반 비즈니스 로직 구조화
→ RAG로 정확한 정보 검색
→ 컴플라이언스 에이전트로 검증

판단 기준:
✅ 받는다: 명확한 비즈니스 룰 존재
          정확도가 생명
          규제 산업 (금융, 의료, 법률)
❌ 거절: 비즈니스 룰 자체가 모호함
          "알아서 잘 해줘" 요구
```

**유형 2: 시스템 파편화 (Integration Problem)**

```
증상:
- 데이터가 여러 시스템에 흩어져 있음
- AI와 기존 시스템 연동 안 됨
- 수작업이 너무 많음

고객 예시:
"CRM, ERP, 이메일, 엑셀에 데이터가 
 다 따로 있어서 AI가 전체 그림을 못 봅니다."

우리 솔루션:
→ 데이터 레이크 구축
→ API 연동 파이프라인
→ 엔드투엔드 자동화

판단 기준:
✅ 받는다: 시스템이 3개 이상
          API 접근 가능
          데이터 오너십 명확
❌ 거절: 레거시 시스템, API 없음
          데이터 정책 불명확
          정치적 이슈 (부서 간 싸움)
```

**유형 3: 데이터 거버넌스 부재 (Data Quality Problem)**

```
증상:
- 데이터가 오래되었거나 부정확함
- 민감 정보 보호 정책 없음
- 데이터 업데이트가 안 됨

고객 예시:
"환자 개인정보(PHI)를 AI가 학습하면 
 HIPAA 위반인데, 어떻게 해야 할지 모르겠어요."

우리 솔루션:
→ 데이터 거버넌스 정책 수립
→ 민감 정보 격리 아키텍처
→ 실시간 데이터 동기화

판단 기준:
✅ 받는다: 규제 준수 필수 (HIPAA, GDPR 등)
          민감 정보 존재
          데이터 품질 개선 의지
❌ 거절: 데이터가 아예 없음
          수집부터 해야 함
          데이터 정책 만들 생각 없음
```

#### 5 Whys 기법: 문제의 근본 원인 찾기

**[VIDEO: 5 Whys로 고객 문제의 진짜 원인 찾기]**

**사례: CS 자동화 요청 고객**

```
고객: "CS 챗봇을 만들고 싶어요"

1st Why: "왜 CS 챗봇이 필요하신가요?"
고객: "CS 직원들이 힘들어해요"

2nd Why: "왜 힘들어하시나요?"
고객: "하루에 1,000건씩 문의가 와요"

3rd Why: "왜 그렇게 많이 오나요?"
고객: "고객이 정보를 찾기 어려워해요"

4th Why: "왜 찾기 어려울까요?"
고객: "FAQ가 카테고리별로 분산되어 있고 검색이 안 돼요"

5th Why: "왜 검색이 안 되나요?"
고객: "레거시 시스템이라 검색 기능 자체가 없어요"

→ 진짜 문제: 챗봇이 아니라 검색!
→ 솔루션: 챗봇 만들기 전에 지식 베이스 구조화 필요
```

**핵심 교훈:**
> 고객이 요청하는 솔루션 ≠ 진짜 필요한 솔루션  
> 항상 5번 "왜?"를 물어서 근본 원인 파악!

## 3. ⏰ Cadence (평가 리듬 및 빈도)

**언제, 얼마나 자주 고객을 평가하고 재평가해야 하는지에 대한 가이드입니다.**

**비유:** 농부가 작물을 주기적으로 관찰하듯이, 고객도 정기적으로 재평가해야 합니다.

**[VIDEO: 고객 평가 주기 - 언제 다시 확인해야 하나?]**

### Always (매 미팅마다)

**CFS 점수 업데이트**

모든 고객 미팅 후 1시간 이내에 CFS를 업데이트하세요.

```
미팅 직후 루틴:
1) 미팅 노트 작성 (10분)
2) CFS 5가지 기준 재평가 (5분)
3) 액션 아이템 결정 (5분)
4) CRM 업데이트 (5분)

총 소요 시간: 25분
→ 이 25분이 잘못된 고객에 몇 달 쓰는 걸 막아줍니다!
```

### First Meeting (첫 미팅)

**필수 체크리스트 (30분 미팅 직후)**

```
□ CFS 점수 계산 완료
□ Tier 분류 (1 / 2 / Watch / Decline)
□ 다음 단계 결정
□ 담당자 배정 (Tier 1/2만)
□ 팀 공유 (슬랙 채널)

30분 안에 처리!
```

### After Proposal (제안서 전달 후)

**1주일 후 팔로업**

```
Day 7 체크포인트:

1) 제안서 확인 여부
   → 안 봤으면: 리마인더 발송
   → 봤으면: 피드백 요청

2) 질문 사항 수집

3) 다음 미팅 일정 잡기
   (최대 2주 이내)
```

**2주일 후 재평가**

```
Day 14 체크포인트:

1) 아직 피드백 없음
   → CFS -10점 (관심도 하락)
   → Watch List로 이동

2) 피드백 있음
   → CFS 유지 또는 증가
   → 계약 단계 진행
```

### Quarterly Review (분기별 검토)

**Watch List 고객 재접촉**

```
Q1, Q2, Q3, Q4 시작 시:

Watch List 전체 리뷰
→ 상황 변화 확인
→ 재접촉 이메일/전화
→ CFS 재평가
→ Tier 재분류
```

**성공 고객 체크인**

```
프로젝트 진행 중인 고객:

분기마다 만족도 조사
→ 문제 없는지 확인
→ 추가 니즈 파악
→ 크로스셀 기회 탐색
```

### Category 1: ICP 부합도 (Fit)

#### [ ] 1. "Tried but Failed" 경험이 있는가?

**확인 방법:**
```
✅ YES:
- 구체적 AI 프로젝트 실패 사례 있음
- 실패 원인을 분석하고 설명할 수 있음
- "이번에는 다르게 하겠다"는 구체적 계획 있음

❌ NO:
- AI 경험 전무
- 막연히 "AI 해보고 싶다"
- 트렌드만 따라가려는 태도
```

**NO일 때 대응:**
→ 교육/웨비나 초대
→ 작은 POC 제안 (리스크 헷지)
→ 6개월 후 재접촉

#### [ ] 2. 명확한 비즈니스 문제가 있는가?

**확인 방법:**
```
✅ YES:
- 문제를 구체적 숫자로 설명 가능
- "이 프로세스가 비효율적" 등 명확한 지적
- 문제 해결의 ROI 계산 가능

❌ NO:
- "뭔가 비효율적인 것 같아요"
- "AI로 뭐 할 수 있을까요?"
- 문제를 찾아달라고 요청
```

**NO일 때 대응:**
→ 문제 정의 워크숍 유료 제안
→ 또는 정중히 거절

#### [ ] 3. Vibers가 해결 가능한 문제인가?

**Vibers가 해결하는 3가지 문제:**
```
✅ YES (이 중 하나):
1) 비즈니스 맥락 이해 부재 (Hallucination)
2) 시스템 파편화 (Integration)
3) 데이터 거버넌스 부재 (Data Quality)

❌ NO:
- 단순 도구 도입 (툴 사면 됨)
- 조직 문화 문제 (컨설팅사 가야 함)
- 순수 ML 모델 개발 (데이터 과학 회사 가야 함)
```

**NO일 때 대응:**
→ 적합한 파트너 추천
→ 정중히 거절


### Category 2: 실행 가능성 (Feasibility)

#### [ ] 4. 의사결정권자에 접근 가능한가?

**확인 방법:**
```
✅ YES:
- 미팅 상대가 C-level 또는 VP급
- 또는 의사결정권자와 1단계 거리
- 의사결정 타임라인 명확 (2주 이내)

❌ NO:
- 담당자급, 의사결정권 전혀 없음
- "위에 보고해야 해요" (누군지 모름)
- 의사결정 프로세스 불명확
```

**NO일 때 대응:**
→ 의사결정권자 미팅 요청
→ 안 되면 거절

#### [ ] 5. 현실적인 예산이 확보되었는가?

**확인 방법:**
```
✅ YES:
- 예산 금액 구체적으로 공개
- 최소 1억 이상 (Vibers 프로젝트 최소 규모)
- 예산 승인 완료 또는 승인 프로세스 명확

❌ NO:
- "얼마 드나요?" (예산 개념 없음)
- "최대한 저렴하게"
- 예산 미정, 승인 프로세스 불명확
```

**NO일 때 대응:**
→ 예산 확보 후 재접촉
→ 또는 작은 POC 제안

#### [ ] 6. 기술 인프라가 준비되었는가?

**확인 방법:**
```
✅ YES (최소 Level 2 이상):
- 클라우드 또는 온프레미스 서버 있음
- 데이터베이스 구축되어 있음
- API 접근 가능 (또는 만들 수 있음)

❌ NO (Level 1):
- 데이터가 엑셀에만 있음
- 인프라 전무
- 데이터 수집부터 필요
```

**NO일 때 대응:**
→ 인프라 구축 먼저 권고
→ 6개월 후 재접촉


### Category 3: 리스크 평가 (Risk)

#### [ ] 7. 타사 비난이 주된 이야기인가?

**확인 방법:**
```
✅ SAFE:
- 과거 실패를 자기 책임으로 인식
- "우리가 AI를 잘 몰랐다"
- "이번에는 이렇게 다르게 하겠다"

❌ DANGER:
- "저번 회사가 망쳤어요"
- "다 남 탓"
- 자기 책임 0%
```

**DANGER일 때:**
→ 🚨 절대 받지 마세요!
→ 이번에도 우리 탓할 확률 99%

#### [ ] 8. 비현실적인 기대를 하고 있지 않은가?

**확인 방법:**
```
✅ SAFE:
- 현실적인 타임라인 (최소 3~6개월)
- 현실적인 정확도 기대 (80~90%)
- 제약 조건 이해하고 있음

❌ DANGER:
- "한 달이면 되죠?"
- "AGI처럼 만들어주세요"
- "100% 정확도 되죠?"
```

**DANGER일 때:**
→ 기대치 조정 미팅 필수
→ 안 되면 거절

#### [ ] 9. 성공 기준이 명확한가?

**확인 방법:**
```
✅ YES:
- 측정 가능한 3~5개 기준
- 예) "응답 시간 50% 단축"
- 예) "정확도 90% 이상"

❌ NO:
- "좋아지면 성공"
- "만족스러우면 성공"
- 추상적, 측정 불가능
```

**NO일 때 대응:**
→ 성공 기준 정의 워크숍 필수
→ 합의 안 되면 절대 시작 금지!


### Category 4: 전략적 가치 (Strategic Value)

#### [ ] 10. 레퍼런스 고객이 될 수 있는가?

**확인 방법:**
```
✅ BONUS (이 중 하나):
- 업계 리더 (삼성, LG 등)
- 새로운 버티컬 진입 기회
- 뛰어난 스토리 (성공 사례로 활용 가능)
- 미디어 노출 가능성

❌ 일반:
- 일반 고객 (나쁜 건 아님)
- 레퍼런스 가치는 낮음
```

**BONUS면:**
→ 가격 할인 고려 가능
→ CEO 직접 케어

#### [ ] 11. 크로스셀/업셀 가능성이 있는가?

**확인 방법:**
```
✅ YES:
- 여러 부서에서 AI 니즈 있음
- 첫 프로젝트는 작지만 확장 계획 있음
- 장기 파트너십 가능성

❌ NO:
- 일회성 프로젝트
- 확장 가능성 제로
```

**1차 미팅 (Discovery):**
```

→ Tried but Failed ✅ (+20점)

AX: "그 문제가 해결되면 어떤 가치가 있을까요?"

고객: "온라인 FC(금융컨설턴트)를 AI로 대체하면 
      연간 50억 비용 절감이 가능합니다. 
      더 중요한 건 고객 경험 개선입니다."

→ 명확한 ROI ✅ (+20점)

AX: "예산과 타임라인은 어떻게 되시나요?"

고객: "3억 책정했고, 6개월 안에 MVP 목표입니다. 
      제가 CTO로서 직접 결정 가능합니다."

→ 예산 확보 + 의사결정권 ✅ (+40점)
```

```

데이터: 상품 DB 체계적으로 관리됨
API: 완비

→ 기술 성숙도 Level 4 ✅ (+15점)
```

**최종 CFS: 95점 → Tier 1**

**결과:**
- 1주일 내 제안서 전달
- CEO 직접 고객 미팅
- 계약 체결
- 성공적인 멀티 에이전트 시스템 구축

**교훈:**

> 이런 고객은 모든 리소스를 집중할 가치가 있다.

## 🎯 마무리: 고객 선별의 본질

**고객 선별이란 무엇인가요?**

간단히 말하면:
```
┌─────────────────────────────────────────┐
│  고객 선별 = 성공의 시작                 │
├─────────────────────────────────────────┤
│                                         │
│  1. 잘못된 고객 = 팀의 에너지 낭비       │
│     (몇 달의 시간, 팀 사기 하락)         │
│                                         │
│  2. 올바른 고객 = 성공 사례              │
│     (레퍼런스, 성장, 신뢰 구축)          │
│                                         │
│  3. 거절의 용기 = 전문성의 증거          │
│     (못하는 걸 안 받는 게 프로다)        │
│                                         │
└─────────────────────────────────────────┘
```

**비유로 이해하기:**

의사를 생각해보세요.
- ❌ 나쁜 의사: 모든 환자를 다 받음 → 치료 실패율 높음
- ✅ 좋은 의사: 자기가 치료 가능한 환자만 받음 → 성공률 높음

AX Architect도 마찬가지입니다!

**실천 3원칙:**

1. **CFS 75점 이상 유지하기**
   - 모든 고객을 CFS로 평가
   - 75점 미만은 원칙적으로 거절

2. **4C Framework 따르기**
   - Camcorder: 신호 포착
   - Course: 프로세스 준수
   - Cadence: 주기적 평가
   - Checklist: 최종 판단

3. **거절의 용기 갖기**
   - 안 맞으면 정중히 거절
   - "그래도..."는 실패의 시작
   - 거절도 전문성이다

**마지막 조언:**

> **"모든 고객을 받지 마세요."**  
> **"올바른 고객만 선택하세요."**  
> **"그것이 진짜 전문가입니다."**
