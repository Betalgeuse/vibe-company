### 이 직무의 Mission
**고객과 개발자 사이의 다리** - FDE는 **Architect Hub**의 도움을 받아 고객 미팅과 스코프 의사결정에만 집중합니다. 회의 녹음, 요구사항 이슈화, 문서 작성 등 반복적 업무는 Architect Hub가 자동화하며, FDE는 오직 고가치 업무(전략적 판단, 고객 관계, 리스크 관리)에만 시간을 씁니다.

### 🔴 Golden Rule: 대표 직접 소통 원칙

**핵심 철학**:
```
프로젝트 성공 = 의사결정권자(대표)의 Commitment

대표가 위임 = 프로젝트 우선순위 하락 = 실패 확정
```

**절대 원칙**:
1. ✅ **모든 미팅은 고객사 대표와 직접**
2. ⛔ **대표 불참 = 미팅 취소 (리스케줄)**
3. 🚨 **대표가 3회 연속 불참 = 계약 종료 통보**
4. ❌ **"팀장이 대신 참석"은 받지 않음**

**Why?**
```
❌ 대표 위임 시:
   담당자는 의사결정권 없음
   → Scope 확정 불가
   → 다시 대표 보고 필요
   → 회의 2번 = 시간 2배

❌ 담당자만 참석:
   프로젝트 우선순위 낮음
   → 피드백 지연 (며칠 소요)
   → 일정 지연 → 비용 증가

❌ 대표 관심 없음:
   비용 아까워함
   → 추가 요청은 많음
   → 결제는 늦음
   → 갈등 발생
```

**계약서 필수 조항**:
```markdown
제X조 (프로젝트 참여 의무)

1. 고객사는 모든 정기 미팅에 대표(CEO) 또는 최종 의사결정권자가 
   반드시 참석해야 한다.

2. 부득이한 사유로 대표가 불참할 경우, 다음 조건을 만족하는 
   대리인을 지정해야 한다:
   - 현장에서 즉시 의사결정 가능한 권한 보유
   - 계약 금액 ±30% 범위 내 추가 승인 권한
   - 대표와 24시간 내 확인 가능

3. 대표 또는 적격 대리인의 3회 연속 불참 시, 
   luerre AI는 일방적으로 계약을 종료할 수 있다.

4. 제3항에 따른 계약 종료 시:
   - 이미 지급된 계약금은 반환하지 않는다
   - 완료된 작업물에 대해서만 인계한다
   - 잔금 청구는 하지 않는다

5. 고객사의 귀책 사유로 인한 계약 종료임을 확인한다.
```

## 📘 R&R (책임과 역할)

### 핵심 책임 (Architect Hub 활용)

Dan Martell의 Buyback Principle: **고가치 업무만 직접 수행, 나머지는 위임/자동화**

| 책임 | FDE 역할 | Architect Hub 역할 | 비중 |
|------|---------------|-------------------|------|
| **고객 미팅** | 미팅 리드, 핵심 질문, 관계 구축 | 자동 녹음, 전사 | 40% |
| **스코프 의사결정** | 우선순위 판단, 추가 요청 수락/거절 | 비용/일정 영향도 자동 계산 | 30% |
| **AI QA 리뷰** | QA 정확성 검증, 누락 확인 | 회의록→QA 자동 생성 | 15% |
| **리스크 관리** | 전략적 리스크 식별, 대응 계획 | 일정/예산 알림 자동화 | 10% |
| **고객 관계** | 신뢰 구축, 갈등 조정 | 진행 상황 대시보드 자동 공유 | 5% |

### Architect Hub에 위임된 업무 (자동화)

**Before (수동)** → **After (Architect Hub)**:
```
❌ 회의록 작성 2시간        → ✅ AI 자동 전사 + QA 생성 5분
❌ Linear 이슈 작성 2시간   → ✅ CACO 자동 변환 + 발행 즉시
❌ 주간 리포트 1시간        → ✅ Progress Bar 자동 업데이트
❌ 고객 문의 응답 1시간     → ✅ Dashboard self-service
❌ 파일 업로드/정리 30분    → ✅ 자동 저장/링크
```

**시간 절감 효과**: 프로젝트당 주 5.5시간 → 2시간 (64% 절감)

### 권한

- 프로젝트 스코프 최종 결정
- 추가 요청 수락/거절 권한
- 고객 미팅 일정 결정
- 견적 승인 (CEO 협의)

## 🎯 Core Mission: Scope Creep 방지

### luerre AI의 핵심 문제 해결

```
┌─────────────────────────────────────────────────────┐
│  외주 개발 실패의 #1 원인: Scope Creep              │
├─────────────────────────────────────────────────────┤
│                                                     │
│  "이것도 추가해주세요"                               │
│  "그건 당연히 포함된 거 아닌가요?"                    │
│  "이건 간단한 수정이잖아요"                          │
│         ↓                                          │
│  프로젝트 범위 2배 → 일정 지연 → 예산 초과           │
│         ↓                                          │
│  고객 불만 + 개발자 번아웃                          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### FDE의 역할

```
┌─────────────────────────────────────────────────────┐
│  FDE = Scope의 수호자 + 대표 소통 전문가            │
├─────────────────────────────────────────────────────┤
│                                                     │
│  대표와 직접 미팅 (위임 받지 않음)                   │

│         ↓                                          │
│  새 요청 = 새 이슈 = 새 공수 산정                   │
│         ↓                                          │
│  투명한 범위 관리 → 분쟁 0건                        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 📝 요구사항 → Linear 이슈 변환

### 이슈 작성 템플릿

```markdown
## 이슈 제목
[기능] 간결한 한 줄 설명

## 배경 (Context)
- 왜 이 기능이 필요한가?
- 어떤 문제를 해결하는가?

## 요구사항 (Requirements)
1. [필수] 첫 번째 요구사항
2. [필수] 두 번째 요구사항
3. [선택] 세 번째 요구사항

## QA 기준 (Acceptance Criteria)
- [ ] 조건 1을 만족하면 성공
- [ ] 조건 2을 만족하면 성공
- [ ] 조건 3을 만족하면 성공

## 레퍼런스
- [참고 사이트 1](URL)
- [참고 사이트 2](URL)

## 예상 공수
- 개발: X시간
- 테스트: X시간
- 총: X시간

## 우선순위
P0 / P1 / P2 / P3

## 고객 확인
- [ ] 고객 QA 확인 완료 (YYYY-MM-DD)
```

### 우선순위 기준

| 우선순위 | 정의 | 예시 |
|---------|------|------|
| **P0** | 이거 없으면 런칭 불가 | 로그인, 결제 |
| **P1** | 런칭에 필요하지만 약간 지연 가능 | 알림, 검색 |
| **P2** | 있으면 좋은 기능 | 다크모드, 필터 |
| **P3** | 나중에 해도 되는 기능 | 공유, 통계 |

### QA 기준 작성 원칙

**좋은 QA 기준**:
```markdown

**3점 추정법**:
```

예시:
- 낙관적: 4시간 (문제없이 진행될 경우)
- 현실적: 8시간 (일반적인 경우)

### 🔴 Golden Rule: 대표 직접 소통 (최우선)
```
1. 모든 미팅은 고객사 대표와 직접
2. 대표 불참 = 미팅 즉시 취소
3. 3회 연속 불참 = 계약 종료
4. "담당자 대신 참석"은 받지 않음
5. Architect Hub가 자동 추적 및 경고

Why? 
- 대표 위임 = 프로젝트 우선순위 하락 = 실패 확정
- 담당자는 의사결정권 없음 = 시간 낭비 2배
```
