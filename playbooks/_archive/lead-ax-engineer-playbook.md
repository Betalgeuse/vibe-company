## 🏠 luerre AI 연결

> **기준 문서**: [company.md](./company.md)

### Vision
**Make people do what they love**
- 고객이 비즈니스에만 집중할 수 있게
- 개발자가 코딩에만 집중할 수 있게

### 이 직무의 Mission
**팀 전체가 코딩에만 집중할 수 있게** - Lead는 Tech Discovery로 최신 기술을 팀에 전파하고, 코드 리뷰와 멘토링으로 주니어가 성장하도록 돕습니다. 팀 전체의 생산성이 곧 Lead의 성과입니다.

### Core Values 실천

| Core Value | 이 직무에서의 실천 |
|------------|-------------------|
| **No ego, Just impact** | 주니어 의견도 경청, 팀 성과가 내 성과 |
| **Don't (just) get shit done** | 새 기술 무조건 도입 X, 임팩트 있는 기술만 |

### Problem 해결 기여

| company.md Problem | Lead AX Engineer 기여 |
|-------------------|----------------------|
| Scope Creep | 기술적 실현 가능성 검토, 과도한 요청 필터링 |
| 불투명한 진행 | 팀 이슈 상태 관리, 리스크 조기 발견 |
| 커뮤니케이션 오버헤드 | 주니어 대신 기술 질문 대응 |

### 팀 철학
우리는 이런 사람을 환영합니다:
- 평생 공부만 했는데 스타트업 씬에 도전해보고 싶은 사람
- The brave dropout - 용기 있게 선택했고, 꾸준히 공부하는 사람
- 은퇴했지만 7살의 마인드셋으로 배우고 싶은 사람

### 새 기술 도입 프로세스

```
┌─────────────────────────────────────────────────────┐
│           Tech Discovery Pipeline                   │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 발견      2. 평가      3. 실험      4. 도입     │
│    🔍    →     ⚖️    →     🧪    →     🚀        │
│    ↓           ↓           ↓           ↓          │
│  #tech-radar  체크리스트   1주 테스트   문서화      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

#### Step 1: 발견 (Discovery)

**트리거**:
- X/스레드에서 흥미로운 도구 발견
- 팀원이 추천
- 반복되는 문제에 대한 해결책 검색

**액션**:
```markdown
## #tech-radar 채널에 공유

**도구명**: [이름]
**링크**: [URL]
**발견 경로**: X / ProductHunt / GitHub / 기타
**한 줄 요약**: [이 도구가 뭔지]
**우리 팀에 적용 가능성**: 높음 / 중간 / 낮음
**관련 문제**: [우리 팀의 어떤 문제를 해결할 수 있는지]
```

#### Step 2: 평가 (Evaluation)

**체크리스트**:

| 기준 | 질문 | 통과 조건 |
|------|------|----------|
| **문제 해결** | 우리 팀의 어떤 문제를 해결하나? | 명확한 Pain Point 존재 |
| **ROI** | 학습 비용 대비 생산성 향상은? | 1주 학습 → 20% 이상 개선 |
| **안정성** | 프로덕션 레디인가? | GitHub Stars 1k+ 또는 기업 사용 사례 |
| **커뮤니티** | 문서/지원이 충분한가? | 공식 문서 + 활발한 커뮤니티 |
| **통합성** | 기존 스택과 호환되는가? | 큰 변경 없이 도입 가능 |

**평가 결과 기록**:
```markdown
## 평가 결과: [도구명]

**날짜**: 2026-01-17
**평가자**: 이상혁

### 점수 (각 항목 1-5점)
- 문제 해결: 5/5
- ROI: 4/5
- 안정성: 4/5
- 커뮤니티: 5/5
- 통합성: 3/5

**총점**: 21/25 ✅ 실험 진행

### 결론
[실험 진행 / 보류 / 기각]
```

#### Step 3: 실험 (Experiment)

**기간**: 1주일

**방법**:
1. 사이드 프로젝트 또는 내부 도구에서 테스트
2. FCSR Before/After 비교
3. 실제 사용 시 장단점 기록

**실험 리포트**:
```markdown
## 실험 결과: [도구명]

**기간**: 2026-01-17 ~ 2026-01-24
**실험자**: 이상혁

### FCSR 비교
- Before: 75%
- After: 85%
- 개선: +10% ✅

### 장점
- [장점 1]
- [장점 2]

### 단점/주의점
- [단점 1]
- [해결 방법]

### 도입 추천 여부
✅ 추천 / ⚠️ 조건부 추천 / ❌ 비추천

### 도입 시 필요한 것
- [ ] 팀 교육 (예상 시간: Xh)
- [ ] 문서 업데이트
- [ ] 기존 코드 마이그레이션 (영향 범위: X)
```

#### Step 4: 도입 (Rollout)

**체크리스트**:
- [ ] Junior AX Engineer Playbook 업데이트
- [ ] Tech Standards 문서 업데이트
- [ ] Tech Sharing 세션에서 발표
- [ ] 팀원 온보딩 완료
- [ ] 모니터링 설정 (문제 발생 시 롤백 계획)

## 👥 팀 멘토링

### 주니어 성장 지원

**목표**: 주니어의 개인 FCSR을 매월 5%씩 향상

**방법**:

#### 1. 페어 프로그래밍 (주 1회, 1시간)
```
┌────────────────────────────────────┐
│  페어 프로그래밍 세션 구조          │
├────────────────────────────────────┤
│  0-10분: 오늘 할 작업 설명         │
│  10-45분: 함께 코딩               │
│  45-60분: 회고 및 피드백           │
└────────────────────────────────────┘
```

#### 2. 코드 리뷰 피드백
- 단순 승인/거절이 아닌 교육적 코멘트
- "왜" 이렇게 하는지 설명
- 더 나은 방법 제안 시 레퍼런스 링크 첨부

**리뷰 코멘트 예시**:
```markdown
❌ 나쁜 예:
"이거 고쳐주세요"

✅ 좋은 예:
"여기서 `useMemo`를 사용하면 리렌더링을 줄일 수 있어요.
참고: https://react.dev/reference/react/useMemo
비슷한 패턴이 `/components/Dashboard.tsx:45`에 있으니 참고하세요!"
```

#### 3. 1-on-1 미팅 (격주, 30분)

**아젠다**:
```markdown
## 1-on-1 with [이름]

**날짜**: 2026-01-17

### 지난 2주 성과
- FCSR: X% (목표 대비 +/-X%)
- 완료 이슈: X개

### 잘한 점
- 

### 개선할 점
- 

### 다음 2주 목표
- 

### 필요한 지원
- 
```

## 🔧 기술 표준 관리

### 책임 범위

| 항목 | Lead의 역할 |
|------|------------|
| **기술 스택** | 최종 결정권 |
| **코딩 컨벤션** | 표준 정의 및 리뷰 |
| **아키텍처** | 설계 및 검토 |
| **인프라** | IaC 정책 수립 |

### 기술 스택 결정 프로세스

```
제안 → 팀 논의 → Lead 결정 → 문서화 → 적용
```

**결정 기준**:
1. 팀의 학습 곡선
2. 프로덕션 안정성
3. 커뮤니티 지원
4. 장기 유지보수성
5. 비용

## 📋 코드 리뷰 가이드

### 리뷰 우선순위

| 우선순위 | 항목 | 예시 |
|---------|------|------|
| **P0** | 보안 | API 키 노출, SQL Injection |
| **P1** | 버그 | 로직 오류, 엣지 케이스 미처리 |
| **P2** | 성능 | 불필요한 리렌더링, N+1 쿼리 |
| **P3** | 가독성 | 네이밍, 주석 |
| **P4** | 스타일 | 포맷팅 (자동화로 해결) |

### 리뷰 SLA

| 상황 | 목표 시간 |
|------|----------|
| 핫픽스 | 1시간 이내 |
| 일반 PR | 4시간 이내 |
| 대규모 변경 | 24시간 이내 |

### 리뷰 체크리스트

```markdown
## PR 리뷰 체크리스트

### 필수 (P0-P1)
- [ ] 보안 이슈 없음
- [ ] 테스트 통과
- [ ] 기존 기능 깨지지 않음

### 권장 (P2-P3)
- [ ] 성능 이슈 없음
- [ ] 코드 가독성 양호
- [ ] 적절한 에러 핸들링

### 선택 (P4)
- [ ] 코딩 컨벤션 준수
- [ ] 불필요한 코드 없음
```

## 🚀 배포 관리

### 배포 승인 권한

**Lead만 할 수 있는 것**:
- Production 배포 승인
- 핫픽스 배포 실행
- 롤백 결정

**주니어도 할 수 있는 것**:
- Staging 배포
- PR 머지 (리뷰 통과 후)

### 배포 체크리스트

```markdown
## Production 배포 체크리스트

**날짜**: 2026-01-17
**배포자**: 이상혁

### 사전 확인
- [ ] 모든 테스트 통과
- [ ] Staging에서 검증 완료
- [ ] 변경 사항 문서화
- [ ] 롤백 계획 준비

### 배포 후
- [ ] 헬스체크 확인
- [ ] 에러 로그 모니터링 (30분)
- [ ] 주요 기능 수동 테스트

### 문제 발생 시
- [ ] 즉시 롤백
- [ ] #incident 채널에 보고
- [ ] 포스트모템 작성
```
