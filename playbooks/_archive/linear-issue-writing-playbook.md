## 🎯 The One Number: ICS (Issue Completeness Score)

Linear 이슈의 완성도를 측정할 단 하나의 지표.

**[VIDEO: ICS 개념 소개 - 왜 이슈 완성도가 중요한가?]**

### 정의 (쉽게 풀어쓰면)

**Issue Completeness Score (이슈 완성도 점수)**

팀원이 이슈를 읽고 **"이거 어떻게 하는 건데요?"라고 물어보지 않는** 비율입니다.

**비유로 이해하기:**
레고 조립 설명서를 생각해보세요.
- ❌ 나쁜 설명서: "레고 성 만들기" + 사진 1장 (어떻게 만들지 모름)
- ✅ 좋은 설명서: 단계별 그림 + 필요한 부품 목록 + 완성 사진 (바로 만들 수 있음)

### 왜 이 숫자가 중요한가?

**1. 팀의 시간을 아껴줍니다**
```
불완전한 이슈              완전한 이슈
     ↓                        ↓
"이거 뭔데요?"            추가 질문 없음
     ↓                        ↓
회의 30분 소집            바로 작업 시작
     ↓                        ↓
실제 작업: 1시간          실제 작업: 1시간
───────────────────────────────────────
총 소요: 1.5시간          총 소요: 1시간
```

**2. FCSR과 직결됩니다**
- 이슈가 완전하면 → 개발자/AI가 한 번에 성공
- 이슈가 불완전하면 → 여러 번 수정, FCSR 하락

**3. 위임의 핵심입니다**
- 내가 없어도 다른 사람이 할 수 있음
- AI에게 맡겨도 제대로 작동함
- 팀 확장이 가능해짐

### 계산 방식 (간단합니다!)

```
ICS = (추가 질문 없이 완료된 이슈) ÷ (전체 이슈) × 100%

예시:
이번 주 20개 이슈를 만들었는데, 16개는 추가 질문 없이 완료
→ ICS = 16 ÷ 20 × 100% = 80% ✅
```

**시각화:**
```
전체 이슈 20개
├─ ✅ 완전한 이슈: 16개 (80%)
│   → 팀원이 바로 작업 시작
│
└─ ❌ 불완전한 이슈: 4개 (20%)
    → "이거 어떻게 하나요?" 질문 옴
    → 추가 설명 필요

목표: 80% 이상 달성하기!
```

**측정 방법:**

매주 금요일에 확인하세요:
```markdown
## 이번 주 ICS 측정

**기간:** 2026-01-13 ~ 2026-01-17
**작성한 이슈:** 20개
**추가 질문 받은 이슈:** 4개
**완전한 이슈:** 16개

**ICS:** 16 ÷ 20 = 80% ✅

### 불완전했던 이슈 분석:
- ISS-123: 디자인 레퍼런스 없었음
- ISS-124: 기술 스택 명시 안 함
- ISS-125: 결과물 파일 경로 빠뜨림
- ISS-126: 제약사항 너무 모호함

### 다음 주 개선 포인트:
- 템플릿 사용하기
- Reference 섹션 필수로 추가
- 5가지 체크리스트 확인 후 발행
```

**목표:** 80% 이상

## 📘 The 4C Framework

Linear 이슈 작성을 4가지 요소로 구조화합니다.

**비유:** 택배 상자를 생각해보세요.
- **Camcorder (보기)**: 상자 겉면에 사진 붙여두기 (이렇게 생겼어요!)
- **Course (설명서)**: 상자 안에 조립 설명서 (이렇게 하세요!)
- **Cadence (시기)**: 언제 작성하고 언제 업데이트하나요?
- **Checklist (확인)**: 발송 전 체크리스트 (빠진 거 없나요?)

```
┌─────────────────────────────────────────────────┐
│         Linear Issue 4C Framework               │
├─────────────────────────────────────────────────┤
│                                                 │
│  1. 🎥 Camcorder  → 보여주기 (예시)             │
│     "완벽한 이슈는 이렇게 생겼어요!"              │
│                                                 │
│  2. 📚 Course     → 작성법 (매뉴얼)              │
│     "4가지 요소를 채우세요"                       │
│                                                 │
│  3. ⏰ Cadence    → 타이밍 (언제)                │
│     "작업 시작 전 + 완료 후 업데이트"             │
│                                                 │
│  4. ✅ Checklist  → 검증 (5가지)                 │
│     "발행하기 전에 확인하세요"                    │
│                                                 │
└─────────────────────────────────────────────────┘
```

## 1. 🎥 Camcorder (이슈는 이렇게 생겼어야 합니다)

완벽한 Linear 이슈의 "전형(典型)"을 보여드립니다.

**왜 예시가 중요한가?**
- 글로 100줄 설명하는 것보다 1개 예시가 더 효과적
- "아, 이렇게 쓰면 되는구나!" 직관적 이해
- 복사해서 바로 쓸 수 있음

**[VIDEO: 완벽한 Linear 이슈 작성 과정 - 처음부터 끝까지]**

### 완벽한 이슈 예시 (Gold Standard)

```markdown
Title: [Auth] 이메일 로그인 페이지 구현

[Context] 배경 설명
──────────────────────────────────────────
현재 상태:
- 회원가입 기능은 구현 완료 (ISS-045)
- 로그인 기능이 없어서 기존 유저가 재접속 불가
- Supabase Auth 이미 세팅됨

왜 필요한가:
- 테스터들이 "로그인이 안 돼요" 피드백 5건 받음
- 유저 리텐션을 위해 필수 기능

관련 이슈:
- ISS-045 (회원가입) 완료됨
- ISS-048 (비밀번호 재설정) 대기 중


[Action] 정확히 무엇을 만들어야 하는가
──────────────────────────────────────────
구현할 기능:
1. 이메일/비밀번호 입력 폼
2. Supabase signInWithPassword() 호출
3. 로그인 성공 시 /dashboard로 리다이렉트
4. 실패 시 에러 메시지 표시 (toast 알림)

사용자 시나리오:
1. 유저가 /login 접속
2. 이메일, 비밀번호 입력
3. "로그인" 버튼 클릭
4. 로딩 스피너 표시 (0.5~2초)
5. 성공 → /dashboard로 이동
6. 실패 → "이메일 또는 비밀번호가 잘못되었습니다" 메시지


[Constraint] 제약사항
──────────────────────────────────────────
디자인:
- Linear.app 로그인 페이지 스타일 참고
- 미니멀한 디자인 (불필요한 요소 없음)
- 폰트: Inter, 컬러: #000000 (텍스트), #5E6AD2 (버튼)

기술 스택:
- Next.js 14 App Router
- Supabase Auth
- TypeScript (필수)
- Tailwind CSS (또는 Inline Style if 안 먹힘)

에러 처리:
- 잘못된 비밀번호: "이메일 또는 비밀번호가 잘못되었습니다"
- 네트워크 에러: "네트워크 연결을 확인해주세요"
- 이메일 미인증: "이메일 인증을 완료해주세요"

보안:
- 비밀번호 입력 필드는 type="password"
- 로그인 실패 시 구체적인 이유 노출 금지 (보안)


[Output] 결과물
──────────────────────────────────────────
생성될 파일:
- app/login/page.tsx (로그인 페이지)
- components/LoginForm.tsx (폼 컴포넌트)
- lib/auth.ts (로그인 로직, 있으면 수정)

테스트 케이스:
- ✅ 올바른 이메일/비밀번호로 로그인 성공
- ✅ 잘못된 비밀번호로 로그인 실패 → 에러 메시지
- ✅ 존재하지 않는 이메일로 로그인 실패 → 에러 메시지
- ✅ 빈 입력값으로 제출 → 폼 validation 에러
- ✅ 로그인 성공 후 /dashboard로 리다이렉트 확인

완료 기준:
- 위 5개 테스트 케이스 모두 통과
- 데스크탑/모바일 모두 정상 작동
- TypeScript 에러 0개


[Reference] 참고 자료
──────────────────────────────────────────
디자인 레퍼런스:
- https://linear.app/login (메인 참고)
- https://vercel.com/login (로딩 상태 참고)

기술 문서:
- https://supabase.com/docs/guides/auth/auth-password
- https://nextjs.org/docs/app/building-your-application/routing/redirecting

관련 코드:
- app/signup/page.tsx (회원가입, 비슷한 구조)


[Estimate] 예상 소요 시간
──────────────────────────────────────────
- 2 points (약 4시간)

[Priority] 우선순위
──────────────────────────────────────────
- High (긴급)

[Labels]
──────────────────────────────────────────
- feature
- auth
- frontend
```

**이 이슈를 받은 개발자의 반응:**
```
✅ "완벽해요! 추가 질문 없이 바로 시작할게요!"
✅ "레퍼런스까지 있어서 디자인 고민 안 해도 되네요"
✅ "테스트 케이스가 명확해서 완료 기준이 분명해요"
```

**이 이슈를 받은 AI의 반응:**
```
✅ Context가 명확해서 배경 이해 완료
✅ Action이 구체적이어서 정확히 무엇을 만들지 알겠음
✅ Constraint가 있어서 엉뚱한 코드 생성 방지
✅ Output이 정의되어 파일 구조 명확
→ FCSR 100% 달성 가능!
```

## 2. 📚 Course (완벽한 이슈 작성법)

**[VIDEO: Linear 이슈 작성 마스터 클래스 - 초보부터 고급까지]**


### Chapter 1: The 4-Element Formula (황금 공식)

Linear 이슈 = 개발자/AI에게 던지는 **명령서**

**핵심 원칙:**
> "이슈 하나를 읽고 추가 질문 없이 바로 작업을 완료할 수 있어야 함"

이것이 바로 **FCSR (First-Command Success Rate) 80% 달성**의 비밀입니다.

#### 4가지 필수 요소

**비유:** 
택시를 탈 때를 생각해보세요.

```
┌─────────────────────────────────────────┐
│  택시 기사에게 목적지 알려주기            │
├─────────────────────────────────────────┤
│                                         │
│  ❌ 나쁜 예:                             │
│  "저기... 시청 쪽으로 가주세요..."        │
│  (어디? 왜? 어떤 길로?)                  │
│                                         │
│  ✅ 좋은 예:                             │
│  "시청역 3번 출구 앞까지,                │
│   강변북로로, 5시까지 도착해야 해요"     │
│  (어디, 왜, 어떻게, 언제 = 명확!)        │
│                                         │
└─────────────────────────────────────────┘
```

Linear 이슈도 마찬가지입니다!

**황금 공식: CACO**

```
┌─────────────────────────────────────────┐
│  Linear Issue = CACO                    │
├─────────────────────────────────────────┤
│                                         │
│  [C] Context    → "왜 필요한가?"         │
│  [A] Action     → "무엇을 만드나?"       │
│  [C] Constraint → "어떤 규칙으로?"       │
│  [O] Output     → "결과물은?"            │
│                                         │
└─────────────────────────────────────────┘
```

#### 1️⃣ [Context] 배경 설명

**질문:** "왜 이 작업이 필요한가요?"

**포함할 내용:**
```markdown
[Context]
──────────────────────────────────────────
현재 상태:
- 지금 무엇이 있고, 무엇이 없는지
- 어떤 문제가 발생하고 있는지

왜 필요한가:
- 이 작업을 안 하면 어떤 문제가 생기는지
- 유저/팀/비즈니스에 어떤 영향이 있는지

관련 이슈:
- 선행 작업 (완료된 것)
- 후속 작업 (대기 중인 것)
```

**예시:**
```markdown
[Context]
현재 상태:
- 회원가입 기능 구현 완료
- 로그인 기능이 없음

왜 필요한가:
- 기존 유저가 재접속 불가
- 테스터 5명이 피드백 남김

관련 이슈:
- ISS-045 (회원가입) 완료
```

**나쁜 예 ❌:**
```markdown
로그인 기능이 필요함
```
→ 왜? 현재 상태는? 배경이 없어서 팀원이 맥락을 모름

#### 2️⃣ [Action] 할 일

**질문:** "정확히 무엇을 만들어야 하나요?"

**포함할 내용:**
```markdown
[Action]
──────────────────────────────────────────
구현할 기능:
1. 첫 번째 기능
2. 두 번째 기능
3. 세 번째 기능

사용자 시나리오:
1. 유저가 A를 한다
2. 시스템이 B를 한다
3. 결과: C가 나온다
```

**핵심:**
- 숫자로 나열 (1, 2, 3...)
- 동사로 시작 ("만들기", "수정하기", "삭제하기")
- 애매한 표현 금지 ("예쁘게", "잘", "적절히" 같은 말 안 쓰기)

**좋은 예 ✅:**
```markdown
[Action]
구현할 기능:
1. 이메일/비밀번호 입력 폼
2. Supabase signInWithPassword() 호출
3. 성공 시 /dashboard 리다이렉트
4. 실패 시 "이메일 또는 비밀번호가 잘못되었습니다" 표시
```

**나쁜 예 ❌:**
```markdown
로그인 페이지 만들기
```
→ 어떤 기능? 무엇을 포함? 너무 모호함

#### 3️⃣ [Constraint] 제약사항

**질문:** "어떤 스타일/기술/규칙으로 만들어야 하나요?"

**포함할 내용:**
```markdown
[Constraint]
──────────────────────────────────────────
디자인:
- 어떤 스타일인지 (미니멀? 화려?)
- 레퍼런스 사이트 (Linear.app처럼)
- 컬러, 폰트 (구체적으로)

기술 스택:
- 프레임워크 (Next.js 14, React 18...)
- 라이브러리 (Supabase, Tailwind...)
- 언어 (TypeScript 필수!)

규칙:
- 에러 처리 방법
- 보안 고려사항
- 성능 요구사항
```

**좋은 예 ✅:**
```markdown
[Constraint]
디자인:
- Linear.app 로그인 페이지 참고
- 미니멀, 폰트: Inter

기술 스택:
- Next.js 14 App Router
- Supabase Auth
- TypeScript 필수

에러 처리:
- 잘못된 비밀번호: "이메일 또는 비밀번호가 잘못되었습니다"
- 네트워크 에러: "네트워크 연결 확인"
```

**나쁜 예 ❌:**
```markdown
예쁘게 만들어주세요
```
→ "예쁘게"가 뭔지? 사람마다 기준이 다름!

#### 4️⃣ [Output] 결과물

**질문:** "어떤 파일을 만들어야 하나요?"

**포함할 내용:**
```markdown
[Output]
──────────────────────────────────────────
생성될 파일:
- 파일 경로와 이름 (app/login/page.tsx)
- 수정될 파일 (있다면)

테스트 케이스:
- ✅ 성공 케이스
- ✅ 실패 케이스
- ✅ 엣지 케이스

완료 기준:
- 무엇이 되어야 완료인지
- 체크리스트 형식으로
```

**좋은 예 ✅:**
```markdown
[Output]
생성될 파일:
- app/login/page.tsx
- components/LoginForm.tsx

테스트 케이스:
```

### Chapter 2: Bad vs Good Examples (실전 비교)

**[VIDEO: 실패하는 이슈 vs 성공하는 이슈 - 3가지 실전 예시]**

"좋은 예"와 "나쁜 예"를 보면서 차이를 체감하세요!

#### 예시 1: Feature 구현 (로그인 페이지)

**❌ 나쁜 이슈:**
```markdown
Title: 로그인 만들기

Description:
로그인 기능 필요함. 예쁘게 만들어주세요.
```

**문제점:**
- ❓ 왜 필요한지 모름 (Context 없음)
- ❓ 어떤 기능인지 모름 (Action 모호)
- ❓ "예쁘게"가 뭔지 모름 (Constraint 없음)
- ❓ 무엇을 만들지 모름 (Output 없음)

**팀원의 반응:**
```
개발자: "이거 어떻게 만들어요? 디자인은요?"
팀장: "음... 그냥 심플하게?"
개발자: "파일 이름은요?"
팀장: "아무거나..."
개발자: "테스트는요?"
팀장: "나중에..."

→ 회의 30분 소집
→ 결과: ICS 0%, FCSR 20% (5번 재작업)
```

**✅ 좋은 이슈:**
```markdown
Title: [Auth] 이메일 로그인 페이지 구현

[Context]
현재 상태:
- 회원가입 기능 구현 완료 (ISS-045)
- 로그인 기능이 없어서 기존 유저 재접속 불가
- Supabase Auth 세팅 완료

왜 필요한가:
- 테스터 5명이 "로그인이 안 돼요" 피드백
- 유저 리텐션을 위한 필수 기능

[Action]
구현할 기능:
1. 이메일/비밀번호 입력 폼
2. Supabase signInWithPassword() 호출
3. 성공 시 /dashboard 리다이렉트
4. 실패 시 에러 메시지 표시

[Constraint]
디자인: Linear.app 로그인 페이지 스타일
기술: Next.js 14 + Supabase + TypeScript
에러: "이메일 또는 비밀번호가 잘못되었습니다"

[Output]
파일: app/login/page.tsx, components/LoginForm.tsx
테스트: 성공/실패/빈값 케이스 모두 통과

[Reference]
- https://linear.app/login
- https://supabase.com/docs/guides/auth/auth-password
```

**팀원의 반응:**
```
개발자: "완벽해요! 바로 시작할게요!"

→ 추가 질문 0개
→ 결과: ICS 100%, FCSR 100% (한 번에 완성)
```

#### 예시 3: Refactoring (코드 정리)

**❌ 나쁜 이슈:**
```markdown
Title: 코드 정리

Description:
코드가 더러우니까 정리해주세요.
```

**문제점:**
- ❓ 어떤 파일? (Context 없음)
- ❓ 무엇을 정리? (Action 모호)
- ❓ 어떤 기준? (Constraint 없음)
- ❓ 결과는? (Output 없음)

**✅ 좋은 이슈:**
```markdown
Title: [Refactor] LoginForm 컴포넌트 분리 및 타입 정리

[Context]
현재 상태:
- app/login/page.tsx 파일이 300줄 (너무 김)
- 폼 로직과 UI가 섞여 있음
- TypeScript any 타입 5곳 사용 중

왜 필요한가:
- 코드 가독성 저하
- 유지보수 어려움
- 타입 안정성 부족

[Action]
리팩토링 목표:
1. LoginForm을 별도 컴포넌트로 분리
2. 폼 로직을 Custom Hook으로 분리
3. any 타입 제거, 명확한 타입 정의

변경 사항:
- app/login/page.tsx (200줄 → 50줄)
- components/LoginForm.tsx (신규, 100줄)
- hooks/useLoginForm.ts (신규, 50줄)
- types/auth.ts (타입 정의 추가)

[Constraint]
기능 동일성 보장:
- 기존 기능 100% 유지
- 회귀 테스트 필수

코드 스타일:
- ESLint/Prettier 통과
- TypeScript strict mode
- 함수형 컴포넌트 사용

[Output]
수정될 파일:
- app/login/page.tsx (축소)
- components/LoginForm.tsx (신규)
- hooks/useLoginForm.ts (신규)
- types/auth.ts (타입 추가)

테스트 케이스:
- ✅ 로그인 성공/실패 정상 작동
- ✅ TypeScript 에러 0개
- ✅ ESLint warning 0개
- ✅ 기존 테스트 코드 모두 통과

[Reference]
- React Hook 패턴: https://react.dev/learn/reusing-logic-with-custom-hooks
- 관련 코드: components/SignupForm.tsx (비슷한 구조)
```

**차이점 요약:**

| 항목 | 나쁜 이슈 | 좋은 이슈 |
|-----|---------|---------|
| **Context** | 없음 | 현재 상태 + 이유 명확 |
| **Action** | "정리해줘" (모호) | 구체적 변경사항 나열 |
| **Constraint** | 없음 | 기능 동일성 + 코드 스타일 |
| **Output** | 없음 | 파일 목록 + 테스트 |
| **질문 받음** | 10개 | 0개 |
| **소요 시간** | 8시간 | 3시간 |
| **ICS** | 0% | 100% |

**템플릿 사용법:**

1. **복사하기**: 위 템플릿 전체 복사
2. **빈칸 채우기**: {중괄호} 안의 내용 채우기
3. **체크하기**: 5가지 체크리스트로 검증
4. **발행하기**: Linear에 등록!

**시간 절약:**
```
템플릿 없이             템플릿 사용
     ↓                      ↓
어떻게 쓰지?            복사 → 빈칸 채우기
     ↓                      ↓
30분 고민              5분 작성
     ↓                      ↓
불완전한 이슈          완전한 이슈
```

#### 2. Priority (우선순위)

**4단계 우선순위:**

```
┌─────────────────────────────────────────┐
│  Priority Level                         │
├─────────────────────────────────────────┤
│                                         │
│  🔴 Urgent (긴급)                        │
│     → 지금 당장! (1시간 내)              │
│     → 서비스 장애, 치명적 버그            │
│     → 예: 결제 안 됨, 로그인 불가         │
│                                         │
│  🟠 High (높음)                          │
│     → 오늘 안에 (24시간 내)              │
│     → 중요한 기능, 많은 유저 영향         │
│     → 예: 핵심 기능 추가, 자주 발생 버그  │
│                                         │
│  🟡 Medium (보통)                        │
│     → 이번 주 안에 (1주일 내)            │
│     → 일반 기능, 소수 유저 영향          │
│     → 예: 부가 기능, 사소한 UI 개선      │
│                                         │
│  🟢 Low (낮음)                           │
│     → 나중에 (1주일 이상)                │
│     → Nice-to-have, 긴급하지 않음        │
│     → 예: 코드 정리, 문서 작업           │
│                                         │
└─────────────────────────────────────────┘
```

**판단 기준:**

| 질문 | Yes → 높은 우선순위 |
|-----|-------------------|
| 서비스가 멈추나요? | 🔴 Urgent |
| 많은 유저가 영향받나요? | 🟠 High |
| 비즈니스에 중요한가요? | 🟠 High |
| 나중에 해도 되나요? | 🟢 Low |

**나쁜 예 ❌:**
```
모든 이슈를 Urgent로 설정
→ 결과: 진짜 긴급한 게 뭔지 모름
```

**좋은 예 ✅:**
```
ISS-101: 결제 안 됨 → 🔴 Urgent
ISS-102: 로그인 기능 추가 → 🟠 High
ISS-103: 버튼 색 변경 → 🟡 Medium
ISS-104: README 업데이트 → 🟢 Low
```

#### 3. Estimate (예상 시간)

**포인트 시스템 (Fibonacci):**

```
┌─────────────────────────────────────────┐
│  Estimate Points                        │
├─────────────────────────────────────────┤
│                                         │
│  1 point = 1~2시간                       │
│     → 아주 간단 (버튼 색 변경)            │
│                                         │
│  2 points = 2~4시간                      │
│     → 간단 (간단한 폼 추가)               │
│                                         │
│  3 points = 4~8시간 (하루)               │
│     → 보통 (로그인 페이지)                │
│                                         │
│  5 points = 1~2일                        │
│     → 복잡 (결제 시스템)                  │
│                                         │
│  8 points = 3~5일                        │
│     → 매우 복잡 (대시보드 전체)           │
│                                         │
│  13 points = 1주일 이상                  │
│     → 너무 큼! → 쪼개세요!               │
│                                         │
└─────────────────────────────────────────┘
```

**예상 시간 판단 기준:**

1. **비슷한 작업 찾기**
   - 과거에 했던 비슷한 작업
   - 걸렸던 시간 참고

2. **불확실성 고려**
   - 모르는 기술? → 2배 시간
   - 처음 해보는 것? → 3배 시간

3. **버퍼 추가**
   - 예상 시간 × 1.5배 (항상 예상보다 오래 걸림)

**예시:**
```markdown
# 로그인 페이지 구현

과거 경험: 회원가입 페이지 = 3시간 걸림
비슷함: 로그인도 비슷한 구조
예상: 3시간
버퍼: 3 × 1.5 = 4.5시간

→ Estimate: 3 points (4~8시간)
```

#### 4. Assignee (담당자)

**지정 원칙:**

```
┌─────────────────────────────────────────┐
│  Assignee 지정 규칙                      │
├─────────────────────────────────────────┤
│                                         │
│  ✅ 명확히 지정하기                       │
│     → "누가 할지 모름" 상태 방지          │
│                                         │
│  ✅ 1명만 지정하기                        │
│     → 책임 소재 명확                     │
│     → 협업 필요하면 Sub-task로 분리      │
│                                         │
│  ✅ 실력에 맞게 배정하기                  │
│     → Junior: 1~3 points               │
│     → Senior: 3~8 points               │
│                                         │
└─────────────────────────────────────────┘
```

**나쁜 예 ❌:**
```
Assignee: 없음 (누가 할지 모름)
Assignee: A, B, C (3명 - 누가 책임자?)
```

**좋은 예 ✅:**
```
Assignee: A (명확히 1명)

협업이 필요하면:
ISS-101: 로그인 UI (Assignee: A - Frontend)
ISS-102: 로그인 API (Assignee: B - Backend)
```

#### 5. Due Date (마감일)

**설정 원칙:**

```
마감일 = Estimate + 버퍼

예시:
- Estimate: 3 points (4~8시간)
- 실제 작업: 1일
- 버퍼: +1일 (예상치 못한 이슈)
- Due Date: 2일 후
```

**주의사항:**
- ❌ 너무 타이트: 스트레스, 품질 저하
- ✅ 적당한 여유: 안정적 개발

#### Linear 기능 조합 예시

```markdown
Title: [Auth] 이메일 로그인 페이지 구현

Labels:
- feature (작업 유형)
- auth (카테고리)
- frontend (팀)
- ready (상태)

Priority: 🟠 High
Estimate: 3 points (4~8시간)
Assignee: @john
Due Date: 2026-01-19

Project: Q1 Auth System
Milestone: MVP Launch
```

**검색 활용:**
```
# 내가 할 일 중 긴급한 것만 보기
assignee:me priority:urgent

# 프론트엔드 팀의 이번 주 할 일
label:frontend due:this-week

# 막혀있는 이슈들
label:blocked
```


### Chapter 5: Issue Sizing (크기 조절하기)

**[VIDEO: 이슈 쪼개기와 합치기 - 적절한 크기 만들기]**

**핵심 원칙:**
> **1 Issue = 1일 이내 완료 가능한 크기**

#### 너무 큰 이슈 (Split 필요)

**징후:**
- Estimate 8 points 이상
- Description이 너무 길음 (1000자 이상)
- "그리고", "또한", "추가로" 같은 말이 많음
- 여러 팀이 필요함 (Frontend + Backend)

**문제점:**
```
큰 이슈 (8 points)
     ↓
작업 중 막힘
     ↓
전체가 Blocked
     ↓
진행 상황 파악 어려움
```

**나쁜 예 ❌:**
```markdown
Title: 로그인/회원가입/비밀번호 찾기 전체 구현

Estimate: 13 points (2주)

Description:
로그인 페이지, 회원가입 페이지, 비밀번호 찾기, 
이메일 인증, OAuth 연동, 프로필 수정...
(너무 많음!)
```

**좋은 예 ✅: 3개로 쪼개기**

```markdown
ISS-101: [Auth] 이메일 회원가입 페이지 구현
Estimate: 3 points
Dependencies: 없음
→ 가장 먼저 시작

ISS-102: [Auth] 이메일 로그인 페이지 구현
Estimate: 3 points
Dependencies: ISS-101 완료 후
→ 회원가입 후 진행

ISS-103: [Auth] 비밀번호 재설정 페이지 구현
Estimate: 2 points
Dependencies: ISS-102 완료 후
→ 로그인 후 진행
```

**쪼개는 방법:**

1. **기능별로 쪼개기**
   ```
   "로그인 + 회원가입" 
   → "로그인" / "회원가입"
   ```

2. **레이어별로 쪼개기**
   ```
   "로그인 전체" 
   → "로그인 UI" / "로그인 API"
   ```

3. **단계별로 쪼개기**
   ```
   "결제 시스템" 
   → "결제 UI" / "결제 연동" / "결제 완료 페이지"
   ```

**시각화:**
```
Before (하나의 큰 이슈)
┌────────────────────────────────────┐
│  Auth System (13 points)           │
│  - 회원가입                         │
│  - 로그인                           │
│  - 비밀번호 찾기                    │
│  - OAuth                           │
│  - 프로필                           │
└────────────────────────────────────┘

After (5개의 작은 이슈)
┌──────────┐ ┌──────────┐ ┌──────────┐
│ ISS-101  │ │ ISS-102  │ │ ISS-103  │
│ 회원가입  │→│ 로그인   │→│ 비번찾기 │
│ 3 points │ │ 3 points │ │ 2 points │
└──────────┘ └──────────┘ └──────────┘
     ↓            ↓
┌──────────┐ ┌──────────┐
│ ISS-104  │ │ ISS-105  │
│ OAuth    │ │ 프로필    │
│ 3 points │ │ 2 points │
└──────────┘ └──────────┘
```

#### 너무 작은 이슈 (Merge 필요)

**징후:**
- Estimate 1 point 미만 (30분 이하)
- Description이 너무 짧음 (100자 미만)
- "버튼 색 바꾸기" 같은 단순 작업

**문제점:**
```
작은 이슈 10개
     ↓
이슈 관리 오버헤드
     ↓
실제 작업 시간 < 이슈 생성/관리 시간
     ↓
비효율
```

**나쁜 예 ❌:**
```markdown
ISS-201: 로그인 버튼 파란색으로 변경 (0.5 point)
ISS-202: 로그인 버튼 텍스트 "로그인"으로 변경 (0.5 point)
ISS-203: 로그인 버튼 호버 효과 추가 (0.5 point)
ISS-204: 로그인 버튼 클릭 효과 추가 (0.5 point)
```

**좋은 예 ✅: 1개로 합치기**
```markdown
ISS-201: [UI] 로그인 버튼 스타일 개선
Estimate: 2 points

Description:
[Action]
1. 색상: 파란색 (#5E6AD2)
2. 텍스트: "로그인"
3. 호버: 약간 어두운 파란색
4. 클릭: Scale 0.98 효과

[Output]
- components/LoginButton.tsx 수정
```

**합치는 기준:**
```
┌─────────────────────────────────────┐
│  이슈 합치기 기준                    │
├─────────────────────────────────────┤
│                                     │
│  같은 파일을 수정한다?               │
│  같은 기능에 속한다?                 │
│  같은 사람이 한다?                   │
│  동시에 완료되어야 한다?             │
│                                     │
│  → 4개 모두 Yes면 합치세요!         │
│                                     │
└─────────────────────────────────────┘
```

#### 적절한 크기 (Goldilocks Zone)

**완벽한 이슈 크기:**

```
┌─────────────────────────────────────┐
│  Goldilocks Zone (적당한 크기)       │
├─────────────────────────────────────┤
│                                     │
│  Estimate: 2~5 points               │
│  소요 시간: 4시간 ~ 1일              │
│  Description: 300~800자             │
│  파일 개수: 2~5개                    │
│                                     │
│  ✅ 하루 안에 완료 가능               │
│  ✅ 진행 상황 파악 쉬움               │
│  ✅ 막혀도 전체 영향 적음             │
│  ✅ PR 리뷰 부담 적음                 │
│                                     │
└─────────────────────────────────────┘
```

**예시:**
```markdown
ISS-301: [Feature] 로그인 페이지 구현
Estimate: 3 points (4~8시간)

[Action]
1. UI 구현 (2시간)
2. Supabase 연동 (2시간)
3. 에러 처리 (2시간)
4. 테스트 (2시간)

총 예상: 8시간 = 1일
→ 적절한 크기! ✅
```


### Chapter 6: Dependencies (의존성 표시하기)

**[VIDEO: Linear에서 이슈 간 의존성 관리하기]**

**왜 중요한가?**
```
의존성 없이                의존성 표시
     ↓                          ↓
ISS-102 먼저 시작         ISS-101 먼저 완료
     ↓                          ↓
ISS-101 대기 중           ISS-102 시작
     ↓                          ↓
시간 낭비                 효율적 진행
```

#### Linear에서 의존성 설정하기

**방법 1: "Blocks" 사용**

```
ISS-101 (회원가입)
   blocks
ISS-102 (로그인)
   blocks
ISS-103 (프로필)
```

**이슈에 표시:**
```markdown
ISS-101: 회원가입 구현
Status: In Progress

────────────────────────────
Blocks: ISS-102, ISS-103
→ 이 작업이 끝나야 다음 작업 시작 가능
```

```markdown
ISS-102: 로그인 구현
Status: Backlog

────────────────────────────
Blocked by: ISS-101
→ ISS-101이 완료되어야 시작 가능
```

**방법 2: Description에 명시**

```markdown
Title: [Auth] 로그인 페이지 구현

[Context]
관련 이슈:
- ISS-101 (회원가입) → 완료됨 ✅
- ISS-103 (프로필) → 대기 중 (이 작업 후 시작)

Dependencies:
- ⬅️ ISS-101 완료 필요 (Supabase Auth 세팅)
- ➡️ ISS-103 시작 가능 (로그인 후 프로필 접근)
```

#### 의존성 패턴

**1. Sequential (순차적)**

```
ISS-101 → ISS-102 → ISS-103

예시:
ISS-101: DB 스키마 설계
   ↓
ISS-102: API 구현
   ↓
ISS-103: Frontend 구현
```

**2. Parallel (병렬)**

```
        ┌─ ISS-102 (Frontend)
ISS-101 ┤
        └─ ISS-103 (Backend)

예시:
ISS-101: 디자인 완료
   ├─→ ISS-102: Frontend 구현 (동시 진행)
   └─→ ISS-103: Backend 구현 (동시 진행)
```

**3. Convergent (합류)**

```
ISS-101 ─┐
ISS-102 ─┤─→ ISS-104
ISS-103 ─┘

예시:
ISS-101: 로그인 API
ISS-102: 회원가입 API
ISS-103: OAuth API
   ↓ (3개 완료 후)
ISS-104: Auth 테스트 전체
```

#### 실전 예시

```markdown
# 프로젝트: Auth System 구축

ISS-101: [Setup] Supabase Auth 세팅
Estimate: 2 points
Dependencies: 없음
Blocks: ISS-102, ISS-103, ISS-104
Status: ✅ Done

ISS-102: [Feature] 회원가입 페이지
Estimate: 3 points
Dependencies: ISS-101
Blocks: ISS-105
Status: ✅ Done

ISS-103: [Feature] 로그인 페이지
Estimate: 3 points
Dependencies: ISS-101
Blocks: ISS-105
Status: 🔄 In Progress

ISS-104: [Feature] 비밀번호 재설정
Estimate: 2 points
Dependencies: ISS-101
Blocks: ISS-105
Status: 📋 Backlog

ISS-105: [Test] Auth 통합 테스트
Estimate: 3 points
Dependencies: ISS-102, ISS-103, ISS-104 (모두 완료)
Status: 📋 Backlog
```

**시각화:**
```
     ISS-101
   (Supabase)
        ↓
    ┌───┼───┐
    ↓   ↓   ↓
ISS-102 103 104
(회원) (로그인)(비번)
    └───┼───┘
        ↓
    ISS-105
   (테스트)
```

**팀원이 보는 화면:**
```
ISS-103 (로그인) - In Progress
├─ ⬅️ Depends on: ISS-101 ✅ Done
└─ ➡️ Blocks: ISS-105 (Waiting)

→ "ISS-101 완료되었으니 시작 가능!"
→ "ISS-105는 내가 끝나야 시작 가능"
```

## 3. ⏰ Cadence (언제 작성하나요)

**언제, 어떤 상황에서 이슈를 작성하고 업데이트하는지에 대한 가이드입니다.**

**비유:** 
일기를 쓰는 것처럼, 작업 전/중/후에 이슈를 업데이트하세요.

**[VIDEO: 이슈 작성 타이밍과 업데이트 주기]**

### 작업 시작 전 (Before Start)

**언제:** 작업에 대해 생각이 명확해졌을 때

**체크리스트:**
```
□ 무엇을 만들지 명확한가?
□ 왜 필요한지 설명 가능한가?
□ 레퍼런스를 찾았는가?
□ 얼마나 걸릴지 예상했는가?
□ 누가 할지 정했는가?

→ 5개 모두 체크되면 이슈 작성!
```

**작성 프로세스:**
```
1단계: 아이디어 메모 (2분)
   ↓
2단계: 레퍼런스 찾기 (5분)
   ↓
3단계: 템플릿에 채우기 (10분)
   ↓
4단계: 5가지 체크리스트 확인 (2분)
   ↓
5단계: 발행! (1분)

총 소요: 20분
```

**나쁜 예 ❌:**
```
"일단 시작하고 이슈는 나중에"
→ 결과: 이슈 없이 작업, 팀원 모름, 기록 없음
```

**좋은 예 ✅:**
```
"작업 전에 이슈 먼저 작성"
→ 결과: 팀원이 내가 뭐 하는지 알고, 나중에 참고 가능
```

### 작업 중 (During Work)

**업데이트 주기:** 의미 있는 진행이 있을 때마다

**업데이트할 내용:**
```markdown
# 댓글로 진행 상황 업데이트

[2026-01-17 10:00]
작업 시작!
- UI 구현 중 (50% 완료)

[2026-01-17 14:00]
문제 발견:
- Supabase Auth 에러 발생

[2026-01-17 16:00]
해결 완료!
- 원인: 환경변수 누락
- 해결: .env.local에 SUPABASE_KEY 추가

```

**업데이트 주기:**
- 하루 1번 이상 (긴 작업)
- 막혔을 때 즉시 (Blocked 상태로 변경)

**나쁜 예 ❌:**

**좋은 예 ✅:**

## 4. ✅ Checklist (성공 기준)

**이슈를 발행하기 전에 반드시 확인해야 할 5가지 항목입니다.**

**비유:** 
비행기 이륙 전 체크리스트처럼, 하나라도 빠뜨리면 안 됩니다!

**[VIDEO: 5가지 체크리스트로 완벽한 이슈 만들기]**

```
┌────────────────────────────────────────────┐
│  완벽한 이슈 = 5가지 모두 ✅                 │
├────────────────────────────────────────────┤
│  하나라도 ❌면 발행하지 마세요!              │
└────────────────────────────────────────────┘
```

### [ ] 1. 배경(Context)이 명확한가?

**질문:**
"이 이슈를 처음 보는 사람이 왜 필요한지 이해할 수 있나요?"

**확인 방법:**
```markdown
[Context] 섹션에 다음이 있는지 확인:

□ 현재 상태 (지금 무엇이 있고 없는지)
□ 왜 필요한가 (이유, 배경)
□ 관련 이슈 (선행/후속 작업)
```

**테스트:**
팀원에게 이슈를 보여주고 물어보세요:
```
"이거 왜 필요한지 알겠어?"

✅ "네, 명확해요!" → Pass
❌ "음... 잘 모르겠는데?" → Fail
```

**나쁜 예 ❌:**
```markdown
[Context]
로그인 기능 필요함
```
→ 왜? 현재 상태는? 불충분!

**좋은 예 ✅:**
```markdown
[Context]
현재 상태:
- 회원가입 기능 구현 완료
- 로그인 기능이 없어서 기존 유저 재접속 불가

왜 필요한가:
- 테스터 5명이 피드백 남김
- 유저 리텐션을 위한 필수 기능

관련 이슈:
- ISS-045 (회원가입) 완료됨
```
→ 명확! ✅

### [ ] 2. 할 일(Action)이 구체적인가?

**질문:**
"이 이슈를 읽고 정확히 무엇을 만들어야 할지 알 수 있나요?"

**확인 방법:**
```markdown
[Action] 섹션에 다음이 있는지 확인:

□ 구현할 기능 (숫자로 나열)
□ 사용자 시나리오 (1, 2, 3...)
□ 애매한 표현 없음 ("예쁘게", "잘", "적절히" 금지)
```

**테스트:**
이슈를 AI에게 복사해서 붙여넣기:
```
"이걸 만들어줘"

✅ AI가 한 번에 완성 → Pass
❌ AI가 "이게 뭔데요?" → Fail
```

**나쁜 예 ❌:**
```markdown
[Action]
로그인 페이지 만들기
```
→ 어떤 기능? 무엇을 포함? 모호함!

**좋은 예 ✅:**
```markdown
[Action]
구현할 기능:
1. 이메일/비밀번호 입력 폼
2. Supabase signInWithPassword() 호출
3. 성공 시 /dashboard 리다이렉트
4. 실패 시 "이메일 또는 비밀번호가 잘못되었습니다" 표시

사용자 시나리오:
1. 유저가 /login 접속
2. 이메일, 비밀번호 입력
3. "로그인" 버튼 클릭
4. 로딩 스피너 (0.5~2초)
5. 성공 → /dashboard 이동
```
→ 구체적! ✅

### [ ] 3. 제약사항(Constraint)이 있는가?

**질문:**
"어떤 스타일, 어떤 기술, 어떤 규칙으로 만들어야 하나요?"

**확인 방법:**
```markdown
[Constraint] 섹션에 다음이 있는지 확인:

□ 디자인 (스타일, 레퍼런스)
□ 기술 스택 (프레임워크, 라이브러리)
□ 에러 처리 (어떤 메시지)
□ 보안/성능 고려사항
```

**테스트:**
2가지 방법으로 만들 수 있다면:
```
방법 A: Tailwind CSS
방법 B: Styled Components

제약사항이 있으면: "Tailwind CSS 사용" → Pass
제약사항이 없으면: 개발자가 헤맴 → Fail
```

**나쁜 예 ❌:**
```markdown
[Constraint]
예쁘게 만들어주세요
```
→ "예쁘게"가 뭔지? 사람마다 다름!

**좋은 예 ✅:**
```markdown
[Constraint]
디자인:
- Linear.app 로그인 페이지 스타일
- 미니멀, 폰트: Inter

기술 스택:
- Next.js 14 App Router
- Supabase Auth
- TypeScript 필수

에러 처리:
- 잘못된 비밀번호: "이메일 또는 비밀번호가 잘못되었습니다"
- 네트워크 에러: "네트워크 연결을 확인해주세요"
```
→ 명확! ✅

### [ ] 4. 결과물(Output)이 정의되었는가?

**질문:**
"어떤 파일을 만들고, 어떻게 테스트하고, 언제 완료인가요?"

**확인 방법:**
```markdown
[Output] 섹션에 다음이 있는지 확인:

□ 생성될/수정될 파일 경로
□ 테스트 케이스 (성공/실패/엣지)
□ 완료 기준 (무엇이 되어야 완료)
```

**테스트:**
작업을 완료한 후:
```
"이거 완료된 거 맞아?"

Output이 명확하면: 즉시 확인 가능 → Pass
Output이 불명확하면: 애매함 → Fail
```

**나쁜 예 ❌:**
```markdown
[Output]
로그인 파일 만들기
```
→ 파일 이름? 경로? 테스트는?

**좋은 예 ✅:**
```markdown
[Output]
생성될 파일:
- app/login/page.tsx
- components/LoginForm.tsx

테스트 케이스:
- ✅ 올바른 이메일/비밀번호로 로그인 성공
- ✅ 잘못된 비밀번호로 실패 → 에러 메시지
- ✅ 빈 입력값 → validation 에러

완료 기준:
- 위 3개 테스트 통과
- TypeScript 에러 0개
- 데스크탑/모바일 모두 정상
```
→ 명확! ✅

### [ ] 5. 레퍼런스(Reference)를 첨부했는가?

**질문:**
"디자인 참고 사이트, 기술 문서, 관련 코드가 있나요?"

**확인 방법:**
```markdown
[Reference] 섹션에 다음이 있는지 확인:

□ 디자인 레퍼런스 (URL)
□ 기술 문서 (공식 문서 링크)
□ 관련 코드 (비슷한 파일)
```

**테스트:**
새로운 팀원이 이슈를 보고:
```
"어떻게 만들어야 할지 알겠어?"

Reference가 있으면: "예시 보고 만들게요!" → Pass
Reference가 없으면: "감이 안 오는데..." → Fail
```

**나쁜 예 ❌:**
```markdown
[Reference]
없음
```
→ 어떻게 만들지 감이 안 잡힘!

**좋은 예 ✅:**
```markdown
[Reference]
디자인 레퍼런스:
- https://linear.app/login (메인 참고)
- https://vercel.com/login (로딩 상태)

기술 문서:
- https://supabase.com/docs/guides/auth/auth-password
- https://nextjs.org/docs/app/building-your-application/routing

관련 코드:
- app/signup/page.tsx (비슷한 구조)
```
→ 완벽! ✅

#### ❌ 1. "예쁘게 만들어주세요"

**문제:**
"예쁘게"가 뭔지? 사람마다 기준이 다릅니다.

**대신:**
"Linear.app 로그인 페이지 스타일로 만들어주세요"
→ 구체적 레퍼런스!

#### ❌ 2. "나중에 알려줄게요"

**문제:**
불완전한 이슈는 작업 시작 불가.

**대신:**
정보가 모두 모일 때까지 이슈 발행 미루기.
→ 완전한 이슈만 발행!

#### ❌ 3. 스크린샷만 덩그러니 올리기

**문제:**
이미지만으로는 무엇을 어떻게 해야 할지 불명확.

**대신:**
이미지 + 설명 함께!
```markdown
[스크린샷]

설명:
- 이미지의 로그인 버튼 스타일 참고
- 색상: #5E6AD2
- 크기: 48px 높이
```

#### ❌ 4. "급합니다!" 같은 감정 표현

**문제:**
감정은 작업에 도움이 안 됩니다.

**대신:**
Priority 설정으로 긴급함 표현.
```markdown
Priority: 🔴 Urgent
Due Date: 오늘 (2026-01-17)
```

#### ❌ 5. 하나의 이슈에 여러 작업 섞기

**문제:**
"로그인 + 회원가입 + 비밀번호 찾기" → 너무 큼!

**대신:**
1 Issue = 1 Task
→ 3개로 쪼개기!

#### ❌ 6. 담당자 지정 안 함

**문제:**
Assignee 없으면 누구도 안 함.

**대신:**
반드시 1명 지정!
```markdown
Assignee: @john
```

#### ❌ 7. 테스트 케이스 없음

**문제:**
언제 완료인지 불명확.

**대신:**
명확한 테스트 케이스!
```markdown
테스트:
- ✅ 로그인 성공
- ✅ 로그인 실패
- ✅ 빈 입력값
```

#### ❌ 8. 의존성 표시 안 함

**문제:**
선행 작업이 있는데 동시 진행 → 낭비!

**대신:**
Dependencies 명시!
```markdown
Dependencies: ISS-101 완료 필요
```

#### ❌ 9. 완료 기준 없음

**문제:**
"이거 완료된 거 맞아?" 논쟁 발생.

**대신:**
명확한 완료 기준!
```markdown
완료 기준:
- TypeScript 에러 0개
- 테스트 3개 통과
- 배포 완료
```

#### ❌ 10. 업데이트 안 함

**문제:**
3일 동안 상태 변화 없음 → 팀원 불안.

**대신:**
매일 1번 이상 업데이트!
```markdown
[2026-01-17] 진행 중 (50% 완료)
[2026-01-18] 문제 발견, 해결 중
[2026-01-19] 완료!
```
